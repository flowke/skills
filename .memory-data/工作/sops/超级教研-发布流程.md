# 超级教研-发布流程

## 适用范围

适用于项目 **超级教研** 的发布协作；其对应仓库为 `ai-ppt`，仓库地址为 `https://git.100tal.com/wangxiao_xueyan_suyang_fe/ai-ppt`。

默认项目协作规则先遵循：`sops/项目默认工作流程.md`

## 本项目发布入口

- 测试发布页：[cloud-test 发布页](https://cloud-test.tal.com/k8s-fe/appManage/appManageCenter/appDetail/imageManage?id=2826932)
- 灰度发布页：[cloud 灰度发布页](https://cloud.tal.com/k8s-fe/appManage/appManageCenter/appDetail/imageManage?id=34527)
- 线上发布页：[cloud 线上发布页](https://cloud.tal.com/k8s-fe/appManage/appManageCenter/releaseDetail?id=33556&cd_id=721588)

## 本项目覆盖默认流程的关键点

### 发测试

- 目标分支：`dev`
- 完成分支合并后，应直接把测试发布页发给用户

### 发灰 / 发灰度

- 目标分支：`gray`
- 完成分支合并后，应直接把灰度发布页发给用户

### 发线上

本项目发线上按 **MR + tag** 方式处理：

1. 创建 MR，而不是直接把代码合并到 `master`
2. 拉取最新 tag
3. 基于最新 tag 计算并升级版本号
4. 推送新 tag
5. 必要时再给出线上发布页

## 已知项目约束

- 用户不能直接 `push master`
- 因此创建 MR 可能需要依赖：
  - GitLab API
  - `glab` / 其他 GitLab CLI
  - GitLab push options
  - 或其他受控自动化脚本
- 当本机仅具备普通 `git push` 能力时，不应默认假设一定能直接创建真实 MR；需要根据当时环境判断

## 与默认工作流的关系

- 没有特别说明时，分支选择、detached 临时 worktree 合并策略、输出规则等，都遵循：`sops/项目默认工作流程.md`
- 如需查看 GitLab 分支合并实现细节，可参考：`sops/GitLab-默认分支合并工作流.md`
