# 项目默认工作流程

## 适用范围

适用于 `工作` 模块下的大多数项目协作场景。

默认原则：**关于项目的工作流程，默认都一样；除非某个项目文档或项目专属 SOP 里另有说明。**

因此，处理具体项目时应遵循以下优先级：

1. 先看该项目是否已有明确的“项目特例”说明。
2. 若有特例，按项目特例执行。
3. 若没有特例，按本文档的默认流程执行。

## 使用前先确认的上下文

在执行任何项目动作前，优先确认：

1. 当前讨论的是哪个项目。
2. 项目绑定的仓库身份是否已锚定。
3. 该项目是否已有独立说明，例如：
   - `topics/projects/<项目名>.md`
   - `sops/<项目名>-发布流程.md`

若项目身份尚未锚定，不要直接把某个项目的仓库、发布页或线上入口套用到另一个项目。

## 默认发布工作流

### 发测试

默认理解为：

1. 若用户未指定分支，则默认使用当前工作分支。
2. 若用户明确指定分支，则使用用户指定分支。
3. 将目标分支合并到 `dev` 分支。
4. 合并完成后，直接把该项目的**测试发布页**给用户，方便继续发布。

### 发灰 / 发灰度

默认理解为：

1. 若用户未指定分支，则默认使用当前工作分支。
2. 若用户明确指定分支，则使用用户指定分支。
3. 将目标分支合并到 `gray` 分支。
4. 合并完成后，直接把该项目的**灰度发布页**给用户，方便继续发布。

### 发线上

默认理解为：

1. 创建 MR，而不是直接把代码合并到 `master`。
2. 拉取最新 tag，确保版本基线正确。
3. 基于最新 tag 计算下一个版本号。
4. 推送新 tag。
5. 给用户返回关键结果与必要链接，例如 MR、tag、线上发布页。

## 默认协作规则

### 分支选择规则

- 用户没指定分支时，默认取当前工作分支。
- 用户指定了分支时，以用户指定为准。

### 输出规则

- 完成“发测试”或“发灰”后，不只说明“已合并”，而是优先直接给出对应发布页链接。
- 完成“发线上”相关准备后，优先返回真实结果，例如：
  - 已创建的 MR 链接
  - 已创建或已推送的 tag
  - 必要的线上发布页链接

### 自动化偏好

- 可以在背后通过脚本完成合并、推送、创建 MR 等动作。
- 但对用户输出时，应优先返回**可直接使用的结果**，而不是只描述内部执行过程。

## 已知通用约束

1. 线上流程默认不是直接 `push master`。
2. 创建 MR 可能需要依赖：
   - GitLab API
   - GitLab CLI（如 `glab`）
   - GitLab push options
   - 或其他受控自动化脚本
3. 当执行环境只具备普通 `git push` 能力时，不应默认假设一定能直接创建真实 MR；需要根据当时环境判断。

## 项目文档应该承载什么

当项目没有特殊流程时，项目文档应主要记录：

- 项目名
- 仓库名 / 仓库地址
- 测试、灰度、线上发布页
- 是否存在项目特例

不必在每个项目文档中重复抄写一整套默认流程。

## 项目特例规则

若某个项目存在以下任一差异，应在该项目文档中单独记录，并视为覆盖默认流程：

- 目标分支不是 `dev` / `gray`
- 线上流程不是 “MR + tag”
- 发布入口与常规项目不同
- 权限或审批链路不同
- 任何仓库、环境、发布系统、标签策略上的差异

## 当前整理约定

当前 `工作` 模块中的项目记忆采用以下结构：

- 通用规则：`sops/项目默认工作流程.md`
- 项目身份与入口：`topics/projects/<项目名>.md`
- 项目专属发布说明：`sops/<项目名>-发布流程.md`

其中，项目专属 SOP 默认只记录“项目特有信息”与“是否有特例”，不再重复整份通用流程。
