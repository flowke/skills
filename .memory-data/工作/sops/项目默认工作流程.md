# 项目默认工作流程

## 适用范围

适用于 `工作` 模块下的大多数项目协作场景。

默认原则：**关于项目的工作流程，默认都一样；除非某个项目文档或项目专属 SOP 里另有说明。**

因此，处理具体项目时应遵循以下优先级：

1. 先看该项目是否已有明确的项目信息和项目差异说明。
2. 若项目文档记录了项目差异，则按差异覆盖本文档的通用默认流程。
3. 若没有项目差异说明，则直接按本文档的通用默认流程执行。

## 使用前先确认的上下文

在执行任何项目动作前，优先确认：

1. 当前讨论的是哪个项目。
2. 项目绑定的仓库身份是否已锚定。
3. 该项目是否已有独立说明，例如：
   - `topics/projects/index.md`

若项目身份尚未锚定，不要直接把某个项目的仓库、发布页或线上入口套用到另一个项目。

## 默认发布工作流

### 发测试

默认理解为：

1. 若用户未指定分支，则默认使用当前工作分支。
2. 若用户明确指定分支，则使用用户指定分支。
3. 对 `git.100tal.com` 上的大多数项目，默认将目标分支合并到 `dev` 分支；除非项目文档另有说明。
4. 合并完成后，直接把该项目的**测试发布页**给用户，方便继续发布。若该项目实际对应多个测试发布页（如主应用 + 子应用），则默认把相关测试发布页**一并展示**。

### 发灰 / 发灰度

默认理解为：

1. 若用户未指定分支，则默认使用当前工作分支。
2. 若用户明确指定分支，则使用用户指定分支。
3. 对 `git.100tal.com` 上的大多数项目，默认将目标分支合并到 `gray` 分支；除非项目文档另有说明。
4. 合并完成后，直接把该项目的**灰度发布页**给用户，方便继续发布。若该项目实际对应多个灰度发布页（如主应用 + 子应用），则默认把相关灰度发布页**一并展示**。

### 发线上

默认理解为：

1. 优先调用通用工具 `tools/gitlab_create_mr.py` 创建**正式 MR**，而不是直接把代码合并到 `master`。
2. 创建 MR 后，把已创建的 MR 链接发送给**默认协作者杜涔涔**。默认消息模板：`涔涔, 帮合并下: {mr_url}, 盼回复`。支持通过工具参数临时覆盖接收人。
3. **第二步之后默认停下来等待同事合并**，不立即继续打 tag。
4. 当用户明确表示“**合并完了，继续**”或同等语义时，再恢复并执行 tag 升级流程。
5. 先 `git fetch origin --prune --tags`，同步远端 tag 与 `origin/master` 状态。
6. 读取最新 tag，并默认按 **patch** 升级下一个版本号。
7. 新 tag 默认格式为：`vX.Y.Z`，并默认打在 `origin/master` 的最新 merge commit 上。
8. tag 说明默认由我根据本次工作内容总结成一句简短话；若用户显式给定文案，则以用户为准。
9. 推送新 tag。
10. 默认直接输出该项目的**线上发布页**；若该项目实际对应多个线上发布页（如主应用 + 子应用），则默认把相关线上发布页**一并展示**；必要时同时附上 MR、tag 等关键结果。

默认协作者锚点：`topics/同事/杜涔涔.md`

## 默认协作规则

### 分支选择规则

- 用户没指定分支时，默认取当前工作分支。
- 用户指定了分支时，以用户指定为准。

### 默认分支合并实现策略

适用于“发测试”“发灰”“把某分支合到 `dev` / `gray`”这类常见操作。

1. **不切换当前工作目录分支**。
2. 先 `git fetch origin`，确保远程分支状态最新。
3. 默认使用 **detached 临时 worktree**，基于 `origin/<target>` 创建临时工作树。
4. 在临时 worktree 中创建临时分支并执行 merge。
5. 合并成功后，执行 `git push origin HEAD:<target>`。
6. 清理临时 worktree。

### 关于已有 target worktree 的默认规则

- 如果目标分支（如 `dev`、`gray`）已经在另一个 worktree 中检出，**默认不直接复用**该 worktree。
- 只有在明确确认以下条件时，才可以考虑复用：
  - worktree 干净，无未提交改动
  - 不处于 merge / rebase / cherry-pick 等中间状态
  - 可以确认该 worktree 当前无人依赖或可安全接管
- 在未确认上述条件前，优先继续使用 detached 临时 worktree。

### 输出规则

- 若一个项目实际对应**多个发布页**（例如主应用、子应用、未来课件等），默认不要只给一个入口，而是把该环境下**相关发布页全部展示出来**。
- 完成“发测试”或“发灰”后，不只说明“已合并”，而是优先直接给出对应发布页链接。
- 发线上的第二步完成后，默认先停下来等待同事合并。
- 当用户说“合并完了，继续”后，再继续执行 tag 与发布页输出。
- 完成“发线上”后，优先直接返回该项目的**线上发布页**。
- 如有必要，再一并附上：
  - 已创建的 MR 链接
  - 已创建或已推送的 tag

### 自动化偏好

- 可以在背后通过脚本完成合并、推送、创建 MR 等动作。
- 推荐工具：`tools/merge_branch_via_detached_worktree.py`
- 如需更细的 GitLab 分支合并实现说明，可参考：`sops/GitLab-默认分支合并工作流.md`
- 如需 tag 升级实现细节，可参考：`sops/GitLab-线上Tag升级工作流.md`
- 但对用户输出时，应优先返回**可直接使用的结果**，而不是只描述内部执行过程。

## 已知通用约束

1. 线上流程默认不是直接 `push master`。
2. 创建 MR 可能需要依赖：
   - GitLab API
   - GitLab CLI（如 `glab`）
   - GitLab push options
   - 或其他受控自动化脚本
3. 当执行环境只具备普通 `git push` 能力时，不应默认假设一定能直接创建真实 MR；需要根据当时环境判断。

## 项目文档应该承载什么

当项目没有特殊流程时，项目文档应主要记录：

- 项目名
- 仓库名 / 仓库地址
- 测试、灰度、线上发布页（若同一项目在同一环境存在多个实际入口，应一并记录）
- 默认协作者 / 对接人（当前统一为杜涔涔，见 `topics/同事/杜涔涔.md`）
- 是否存在项目差异

不必为每个项目再复制一份发布流程 SOP。

## 项目差异规则

若某个项目存在以下任一差异，应在项目索引或其他项目差异说明中单独记录，并视为覆盖本文档的通用默认流程：

- 目标分支不是 `dev` / `gray`
- 线上流程不是 “MR + tag”
- 发布入口与常规项目不同
- 权限或审批链路不同
- 任何仓库、环境、发布系统、标签策略上的差异

## 当前整理约定

当前 `工作` 模块中的项目记忆采用以下结构：

- 通用规则总入口：`sops/项目默认工作流程.md`
- GitLab 分支合并实现细节：`sops/GitLab-默认分支合并工作流.md`
- GitLab MR API 创建实现：`sops/GitLab-MR-API自动创建工作流.md`
- 项目身份与入口：`topics/projects/index.md`

也就是说：

- 发布流程默认只保留一份通用 SOP；
- 项目只记录身份信息、入口和差异；
- 只有当某个项目确实出现稳定差异时，才考虑补充额外说明，而不是默认一项目一份 SOP。
