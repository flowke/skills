# 内容生产平台-发布流程

## 适用范围

适用于项目 **内容生产平台** 的发布协作；其对应仓库为 `edu-content-studio`，仓库地址为 `https://git.100tal.com/wangxiao_xueyan_suyang_fe/edu-content-studio.git`。

## 发测试

### 目标

把目标分支合并到 `dev` 分支，用于测试环境发布。

### 默认规则

1. 若用户未指定分支，则默认使用当前工作分支。
2. 若用户明确指定分支，则使用用户指定分支。
3. 将目标分支合并到 `dev` 分支。
4. 合并完成后，直接把测试发布页展示给用户，方便点击打开并继续发布。

### 执行偏好

- 可以在背后通过脚本处理合并动作。
- 输出时优先附上测试发布页链接，而不仅仅说明“已合并”。

## 发灰 / 发灰度

### 执行偏好

## 发线上

### 目标

通过 MR + tag 的方式完成线上发布准备。

### 默认流程

1. **创建 MR**
   - 不是直接合并到 `master`。
   - 应优先创建一个真实存在的 MR，而不是只给预填表单链接。
2. **拉取最新 tag**
   - 需要先同步远程 tag 状态。
3. **升级 tag 版本**
   - 基于最新 tag 计算下一个版本号。
4. **推送新 tag**
   - 将新 tag 推送到远程。
5. **必要时给出线上发布页**
   - 方便后续查看线上发布状态。

### 已知约束

- 用户不能直接 `push master`。
- 因此线上流程可能需要：
  - GitLab API
  - `glab` / 其他 GitLab CLI
  - 或 GitLab push options / 受控自动化脚本
- 当本机仅具备普通 `git push` 能力时，不应默认假设一定能直接创建 MR；需要按当时环境判断是否可通过 API/CLI 或 push option 完成。

## 与用户协作时的默认行为

### 当用户说“发测试”

默认理解为：

- 取当前工作分支（或用户指定分支）
- 合并到 `dev`
- 然后把测试发布页发给用户

### 当用户说“发灰”或“发灰度”

默认理解为：

- 取当前工作分支（或用户指定分支）
- 合并到 `gray`
- 然后把灰度发布页发给用户

### 当用户说“发线上”

默认理解为：

- 创建 MR
- 获取并升级最新 tag
- 推送新 tag
- 再把关键信息和必要链接给用户

## 后续可补强的自动化方向

1. 写一个统一脚本处理：
   - 读取当前仓库状态与工作分支
   - 合并到 `dev` / `gray`
   - 输出对应发布页链接
2. 再补一个线上脚本处理：
   - 检查仓库 `edu-content-studio` 的 GitLab remote
   - 尝试创建真实 MR
   - 获取最新 tag 并升级
   - 推送 tag
3. 若 GitLab API 凭据完善，可把“创建 MR”从“给链接”升级为“直接返回已创建 MR 链接”。
