## 当前真相
- 讨论主题：把 `discuss-and-distill` 升级并迁移为新的 canonical skill identity。
- 当前目标：把它从偏“模糊话题讨论收敛”的 skill，升级为一个以 truth 为核心、可编排 `intake / modules / topics / tasks / knowledge` 的工作编排 skill。
- 核心定位：新的 canonical identity 是 `truth-work-orchestrator`；旧名 `discuss-and-distill` 已完成历史迁移。
- 职责范围（当前已确认）：
  - 接收复杂需求
  - 引导对齐
  - 补遗漏 / 查冲突
  - 在需求稳定后衔接到代码落地规划
  - 覆盖到代码完成后的回归阶段
  - 在代码完成后的回归阶段维护同一份 current truth
  - 根据回归结果更新问题、偏差与后续待决项
- 轻重分流的判断本身是成立的：
  - 轻量需求：更适合直接走 agent plan / 执行流
  - 复杂需求：更适合进入 discuss 这类讨论与沉淀流程
- 但已确认：**skill 本身不负责做轻重分流**。
  - 理由：一旦进入这个 skill，就默认当前已经是值得讨论、对齐和沉淀的场景；分流属于 skill 外部的使用判断，而不是 skill 内部第一步。
- 已确认需要有“**一致性复盘 / 规划前检查**”这一环：
  - 至少在进入代码落地规划之前必须发生
  - 作用是把“看起来讲清楚了”变成“足够稳，可以规划了”
  - **默认作为显式环节保留，但允许按场景轻量化**
  - 无论轻量还是完整，**都不能跳过**
- 这轮一致性复盘至少要覆盖：
  - 冲突检查
  - 相互影响检查
  - 遗漏项检查
  - 待决策点检查
- 已确认“**回归**”的定义：
  - 指**代码写完之后进行回归**
  - 不指广义的“重新进入需求对齐”或“任意阶段回看”
  - 理由：这里要保留工程语义，避免把“回归”扩成泛化的返场机制
- 已确认这个 skill 的职责边界：
  - **覆盖到代码完成后的回归阶段**，否则会与前面的对齐 / 规划阶段形成割裂
  - 但重点是**回归结果的沉淀、问题回写与 truth 闭环**
  - **不负责回归执行本身**，也不替代测试 / 验收动作
- 已确认“**背景资料提交**”的定位：
  - **不单列为固定主流程阶段**
  - 更适合作为进入需求对齐前或对齐过程中**按需触发的输入 / 补充动作**
  - 理由：它更像资料准备与上下文补齐，不是一个必须独立推进的收敛阶段
- 已确认完整主流程的设计方式：
  - 采用**标准骨架 + 按场景裁剪**，而不是每次都要求完整经过固定全链路
  - 推荐主骨架为：**需求对齐 → 一致性复盘 → 代码落地规划 → 回归结果沉淀 → 持续沉淀**
  - 其中“背景资料补充”是按需动作，不单列为固定主阶段
  - 其中“回归结果沉淀”发生在**代码完成后的回归阶段**，负责把结果、问题、偏差回写到同一份 current truth

  - 理由：这两类覆盖当前最主要的执行意图；而“直接创建”应从类型层移出
- 已确认第一版挂接方式至少包含：
  - `独立创建`
  - `挂模块`
  - `挂主题`
  - `挂子模块`
- 已确认 task 目录命名规则采用：`YYMMDD-<中文任务名>/`。
  - 目录名优先承接启动日期与中文任务名
  - `任务类型` 与 `挂接方式` 不编码进目录名，而放入 task 的内容字段中
  - 理由：保持目录可读、简短、稳定，避免把多维语义压成过重命名
- 已确认每个 task 目录至少包含一个入口文件：`task.md`。
- 已确认 `task.md` 的最小字段收敛为：
  - `任务类型`
  - `挂接对象`
  - `当前状态`
- 已确认：`挂接方式` **不再作为最小必选字段**。
  - 理由：在大多数场景下，它可以由 `挂接对象` 推导；如果没有挂接对象，则可理解为任务当前处于独立创建状态
- 已确认当前先不把“来源 intake 项”列为必选字段。
  - 理由：不是所有 task 都一定来自 intake；部分 task 可以直接创建
- 已确认：`来源 intake 项` 正式作为 task 的**可选字段**。
  - 理由：它可以保留从 intake 到 task 的追溯链，但不强迫所有 task 都必须经过 intake
- 已出现一个新的上游需求：**待处理队列**。
  - 用户希望存在一个“待处理队列”功能，用于承接一些尚未正式进入 task 的原始资料
  - 这些资料可能包括：文字描述、文件、URL 等
  - 用户可能会先说“把这些资料加入待处理”，之后在某个时刻再说“开始处理 xxx”，此时才正式新建 task
  - 这意味着在 task 之前，还需要一个**资料接收 / 待处理 / 待启动**的前置层
- 已确认：**待处理队列是独立顶层产物**。
  - 它不属于 task 生命周期内部的一个隐藏阶段，而是位于 task 之前的独立承接层
  - 理由：`加入待处理` 与 `正式创建 task` 是两个不同动作，不应混成一个对象生命周期里的同一步
- 已确认待处理队列的顶层目录名采用：`intake/`。
  - 理由：它比 `queue/` / `pending/` 更能表达“原始资料接收 + 待启动”的语义
- 已确认：`intake/` 下的每个待处理项是**目录级对象**。
  - 理由：待处理项可能同时承接文字描述、文件、URL 等多种原始资料，目录比单文件更稳
- 已确认 intake 项的最小结构采用一个极轻版本：
  - `note.md`
  - `attachments/`
  - 理由：先满足“简短说明 + 原始资料承载位置”即可，避免过早把 intake 做成复杂任务系统
- 已确认 task 的层级关系采用“**父 task 为目录，子 task 为文件**”的轻量模型：
  - 父任务 / 子任务首先是**逻辑上的任务编排关系**
  - 父 task 仍然是目录级对象
  - 子 task 不单独创建目录，而在父 task 目录内以带前缀的文件表达
- 已确认子 task 文件的命名方式：
  - 统一使用 `subtask-` 前缀
  - 当前表达形态例如：`subtask-xxx.md`
- 已确认一条 task 层级设计原则：
  - 目录结构服务于可读性与组织性
  - 子任务不再通过目录嵌套表达，以避免 task 结构过重
- 已确认：`subtask-*.md` 沿用与 `task.md` 相同的最小字段集合。
  - 最小字段同样为：`任务类型` / `挂接对象` / `当前状态`
  - 理由：父 task 与子 task 保持同一最小模型，后续维护与理解成本更低
- 已出现另一个新的 task 结构需求：**task 可以有子任务**。
  - 这意味着 task 不一定总是叶子执行单元，它也可能成为任务编排与拆解的父节点
  - 在某些场景下，一个 task 在对齐资料后，可能拆出多个子任务分别处理
- 已出现一个新的 task 生命周期需求：
  - 某些任务会先经历**资料对齐 / 需求对齐**
  - 当需求对齐完成后，任务才进入**代码规划和执行阶段**
  - 这意味着 task 不只是“已开始执行”的对象，也可能先处在 intake / 对齐阶段
- 用户给出的场景样例：
  - 在 UI 走查场景中，用户可能先把包含多个 UI 问题的原始表格交给 agent，先加入待处理
  - 当用户明确说“处理某个待处理任务”时，再正式创建 task
  - task 创建后先做资料与需求对齐，再根据需要拆成多个子任务，最后进入代码规划与执行
- 已确认 drift handling 采用轻量三分法：
  - **truth 过时**：代码合理，但 truth 没有同步更新
  - **code 偏离**：truth 仍然成立，但代码跑偏了
  - **暂时无法判断**：当前证据不足，不能立即判断该改 truth 还是改 code
- 已确认三类 drift 的默认动作：
  - `truth 过时` → **更新 truth**，不直接改代码
  - `code 偏离` → **修正代码**，不直接改 truth
  - `暂时无法判断` → **挂起为待确认冲突**，不自动改 truth，也不自动宣判代码错误
- 已确认一条鲁棒性原则：
  - **先分类，再动作**
  - 在证据不足时，不自动回写 truth，也不自动修正代码
- 已确认代码落地 task 需要一套独立的标准流程骨架：
  - **创建与挂接 → 任务对齐 → 一致性复盘 → 实现规划 → 执行承接 → 执行中回填 → 完成收口 → 回写与后续交接**
  - 理由：当前 task 的类型、挂接方式与最小字段已经有了，但如果没有流程骨架，就很难稳定承接“从对齐到交接”的连续工作
- 已确认代码落地 task 与回归测试 task 的边界：
  - **代码落地 task** 负责承接一次实现活动
  - **回归测试 task** 负责代码完成后的验证与 drift handling
  - 理由：避免在同一个 task 中混写实现推进与回归结论，重新滑回旧的阶段文件模型
- 已确认代码落地 task 的文档集合采用“**一个强必有主入口 + 若干按需文档**”模型：
  - **强必有**：`task.md`
  - **按需补充**：`implementation-plan.md`、`subtask-*.md`、`handoff.md`、`attachments/`
  - 理由：既保留 task 目录轻量性，又让复杂任务在需要时有明确承接位置
- 已确认代码落地 task 的规划文档如果需要保留，应放在 **task 目录内部**：
  - 推荐文件名：`implementation-plan.md`
  - 它用于承接较复杂代码落地 task 的实现拆解、顺序、风险与依赖
  - 理由：模块 / 主题层的 `planning.md` 已退役，规划职责应归到执行对象层
- 已确认代码落地 task 在需要显式交接时，可以使用 `handoff.md`：
  - 适用于交接给执行 agent / 执行人 / 后续回归测试 task
  - 理由：`task.md` 负责维护主入口真相，`handoff.md` 负责降低跨人或跨轮接手成本
- 已确认代码落地 task 的推荐状态集合：
  - `待对齐`
  - `对齐中`
  - `待规划`
  - `规划中`
  - `待执行`
  - `执行中`
  - `待回写`
  - `已完成`
  - `已挂起`
  - 理由：避免 task 状态只剩“进行中 / 已完成”两档而失去可编排性
- 已完成一轮 reference 补齐：
  - 新增 `references/code-delivery-task-flow.md`，承接代码落地 task 的流程骨架、状态建议、文档集合与回写规则
  - `references/object-templates.md` 已补充代码落地 task 的推荐扩展骨架，以及 `implementation-plan.md` / `handoff.md` 模板
  - `SKILL.md` 已补充该 reference 的读取入口
  - `references/output-model.md` 已补充代码落地 task 与规划 / 回归边界的说明
- 已明确当前完善代码落地 task 流程的核心目标：
  - **规划实施代码时，应优先选择最优落地路径，并尽量减少返工**
  - **过程必须可记录、可中断、可重启**；即使中途切换电脑，也应能通过 task 文档恢复进度
  - 理由：这比单纯补更多模板更关键；流程设计应服务于“更稳地做对”和“中断后能续上”
- 已确认代码落地 task 的路径选择原则：
  - 优先选择 **truth 对齐成本最低、改动面最小、依赖链最短、验证路径最短、回滚成本更低** 的实施路径
  - 如果多条路径都可行，默认优先更少模块联动、更少隐含假设、且更容易阶段性验证的路径
- 已确认代码落地 task 的文档需要具备“续推能力”：
  - `task.md` 至少应能回答：**现在在做什么、为什么这么做、已经做到哪里、下一步做什么**
  - 对于中大型任务，在暂停或切换设备前，建议补 `handoff.md`，明确 `重启入口`
  - 理由：文档应默认服务于“另一台电脑上的自己”也能接上，而不是只服务当前会话
- 已同步补齐模板：
  - 代码落地 task 的 `task.md` 模板已补充 `实施路径选择` 与 `当前进展` 区块
  - `handoff.md` 模板已补充 `重启入口` 字段
- 已确认代码实施中的验证与回归，默认应包含在同一个代码落地 task 内：
  - **实现 → 验证 → 修正 → 再验证 → 回写** 应作为同一个 task 的默认闭环
  - 这种情况**不需要再新起 task**
  - 理由：对多数代码实施场景来说，验证与回归本来就是实施过程的一部分；强行拆新 task 会增加切换成本与文档分裂
- 已相应修正代码落地 task 与回归测试 task 的边界：
  - 代码落地 task 不再默认把验证 / 回归外包给独立 task
  - 独立 `回归测试` task 仅在验证目标明显独立、范围明显外扩时按需使用
- 已补充代码落地 task 的验证文档位：
  - `task.md` 主入口中新增 `验证与回归` 区块
  - 当验证 / 回归记录较复杂时，可按需新增 `verification.md`
- 已确认需要为独立 `回归测试` task 单独补一份专项 flow reference：
  - 理由：当前它虽然已有对象类型与 drift handling，但还缺少“何时起、怎么做、维护哪些文档、如何恢复”的成体系规则
- 已新增 `references/regression-task-flow.md`：
  - 用于定义独立回归任务的定位、流程骨架、文档集合与中断恢复规则
- 已确认独立回归任务的心智：
  - 它是一个**验证型 task**，不负责实施新代码
  - 它负责判断当前代码是否仍与 truth 一致，并给出漂移分类与后续动作
- 已确认独立回归任务与代码落地 task 的边界：
  - 如果验证只是当前代码实施闭环的一部分，应留在代码落地 task 内完成
  - 如果验证目标已独立、范围明显外扩，才新起独立 `回归测试` task
- 已确认独立回归任务默认也支持中断后恢复：
  - 恢复时默认顺序：`task.md` → `verification.md` → `handoff.md`（如果有）
  - 理由：独立回归任务也属于 task，对跨设备 / 跨会话续推的要求应一致
- 已同步补齐对象模板：
  - `references/object-templates.md` 新增了独立 `回归测试` task 的推荐扩展骨架

## 冲突与影响检查
- 原有“主题 + planning/regression”模型已被新的“模块 + 主题 + task + 知识”模型替代，当前不再作为默认结构继续生效。
- 已校正一个关键理解：
  - task 不再是最早进入系统的工作单元。
  - 在 task 之前，存在一个独立的待处理队列层，用于承接原始资料与待启动事项。
- 新出现一个需要继续收敛的点：
  - 之前讨论 task 时，更偏单体任务。
  - 现在用户明确要求 task 可以有子任务，因此 task 需要兼容编排 / 拆解能力，而不只是单节点执行。
- 新出现一个需要继续收敛的点：
  - 之前曾把“当前第一类 task”收敛为代码落地 task。
  - 现在用户补充了更多 task 想法，说明这个判断应降级为**过渡性理解**，而不是最终 task 类型边界。
- 已识别到一个容易混淆的点：
  - “轻重分流成立” 与 “skill 要不要负责分流” 是两回事。
  - 当前已明确：前者成立，后者不成立。
- 已识别到一个关键设计点并已收敛：
  - “一致性复盘”不能缺席。
  - 在交互上默认显式保留，但允许按场景轻量化，以兼顾稳定性与自然度。
