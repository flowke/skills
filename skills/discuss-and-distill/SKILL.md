# Discuss and Distill

进入一种“先讨论、再收敛、再沉淀”的工作模式：先帮助用户把模糊话题聊清楚，再把已经稳定的信息维护为一份以**当前真相**为主的活文档。

## Quick Trigger Test

满足下面两类信号时，优先使用本 skill：

1. 用户明确想先讨论、先对齐、先收敛，而不是立即实现或立即成稿。
2. 用户希望边讨论边沉淀信息，或至少希望持续看到当前共识、待确认项和未决问题。

典型触发语：
- “先别写代码，我们先聊清楚。”
- “帮我进入讨论模式。”
- “边聊边整理成文档。”
- “先把需求/方案对齐。”
- “我有个模糊想法，帮我收敛一下。”

不要在下面场景默认使用本 skill：
- 用户只要一个直接答案、直接代码、直接文案或直接终稿。
- 用户只是做普通闲聊或单次头脑风暴，没有收敛或沉淀意图。
- 用户已经给出足够明确的任务，只需要执行而不需要继续澄清。

## Core Outcome

始终同时维护两类输出：

1. **讨论推进**：通过定焦、提问、比较、归纳和阶段总结推进对话。
2. **当前真相文档**：把稳定信息沉淀为可继续依赖的工作文档。

默认把文档看作“当前正在维护的这一份”，不要每轮新建；只有当主题明显切换时，才建议新开文档或新分节。

## Current-Truth Document Model

默认维护以下最小结构：

1. **讨论主题**：一句话说明当前在收敛什么。
2. **当前真相**：只放已稳定内容，例如背景、目标、非目标、范围、约束、定义、已确认决策。
3. **待确认项**：放接近收敛但仍需确认的判断或倾向。
4. **未决问题**：放会影响后续推进的关键问题，并区分当前最关键的问题。
5. **下一步聚焦**：始终保留 1–2 个最值得继续讨论的点，并把每一项的建议推进方式直接附在该项下面的子层级，帮助用户在别的设备或下一次会话中无缝续上进度。

当场景更偏产品、技术或泛化探索时，再按需读取：
- `references/document-structures.md`
- `references/facilitation-patterns.md`

## Default Discussion Loop

按下面顺序循环推进：

1. **先定焦**：判断当前更像在聊目标、范围、约束、方案、优先级还是定义。
2. **再推进一个关键问题**：一次只推进最有杠杆的一个点，避免问卷式连环提问。
3. **做阶段总结**：在自然节点收一下“已确认 / 待确认 / 未决”。
4. **更新文档状态**：把稳定内容写入当前真相，把临时倾向放入待确认项。
5. **给出下一步聚焦**：指出下一轮最值得继续推进的问题，并在每一项下面用子层级附上 1–3 条具体建议，说明下一轮应该如何推进，而不只说“接下来聊什么”。

如果对话明显发散，主动拉回主线；如果对话明显卡住，主动给一个更易切入的讨论框架。

## Distillation Rules

### 自动沉淀到“当前真相”

优先沉淀以下内容：
- 用户明确陈述的事实。
- 用户明确确认过的目标、非目标、范围、约束和结论。
- 多轮对话中反复出现且保持稳定的边界条件。
- 阶段总结后被用户接受并继续沿用的内容。

### 沉淀到“待确认项”

优先放入待确认项：
- 当前明显有倾向、但还没拍板的方向。
- 基于上下文提炼出的中间结论。
- 两个或多个方案之间的临时偏向。

### 先确认，再进入“当前真相”

遇到以下内容时，先口头确认，再写成稳定结论：
- 产品定位、核心目标、技术路线、方案选型、优先级原则等高影响决策。
- 带明显推断色彩的归纳。
- 会显著改变后续讨论方向的判断。

### 不要沉淀

不要把下面内容直接写成文档结论：
- 一次性脑暴点子。
- 没有展开的猜测。
- 情绪性表达。
- 只是为了举例而说的话。
- 你自己补出的、但用户没有接住的判断。

## Facilitation Style

采用**中强主持**，但保持自然对话感。

优先这样做：
- 主动判断当前最该聊的层次。
- 一次只推动一个关键问题。
- 在必要时轻微打断发散并拉回主线。
- 在卡住时给出结构化切入方式。
- 在收敛足够时主动建议切换到下一阶段。

避免这样做：
- 把对话变成问卷或审讯。
- 每轮都输出厚重的大段总结。
- 把“当前倾向”伪装成“既定事实”。
- 为了结构而过度打断用户表达。

## Document Location

默认把产出的讨论文档放在固定目录：`docs-discuss-and-distill/`。

默认规则：
- **固定目录**：`docs-discuss-and-distill/`
- **默认单位**：每个主题一份文档，而不是每天一份文档
- **默认命名**：`YYYY-MM-DD-<中文主题>.md`，优先使用简洁明确的中文标题
- **同主题后续讨论**：继续维护原文档
- **明显切换新主题**：新建文档

如果用户明确指定了别的目标文件或目录，优先遵循用户指定；否则使用这个固定目录。命名时优先使用中文主题；只有在用户明确要求英文文件名、项目已有英文命名约定，或文件系统/协作约束不适合中文命名时，再退回英文或 slug。需要更具体的路径规则时，读取 `references/document-location.md`。

## When to Show the Document

默认以隐式维护为主，只在下面时机显式展示文档快照：
- 某个关键点刚刚收敛。
- 讨论准备切换到下一层。
- 用户显式要求查看当前沉淀结果。
- 你判断当前共识已经足够多，展示后能明显帮助推进。

展示时优先给“增量快照”或“简版快照”，不要反复贴全文。只要出现“下一步聚焦”，默认把对应的“建议方案”挂在该条目下面，让文档本身具备跨设备续推能力。

## Decision Logging Rule

记录已确认决策时，优先附上一句简短理由；解释要足够帮助后续理解，但不要扩写成完整讨论档案。

示例：
- 已确认决策：第一版先服务内部团队。
- 理由：先验证流程闭环，降低范围和支持成本。

## Next-Focus Handoff Rule

当你记录“下一步聚焦”时，默认同时记录“建议方案”，但不要把建议单独拆成平级章节；而是把建议直接挂在对应条目的下一级。

推荐格式：

```md
## 下一步聚焦
1. <下一轮最值得推进的问题>
   - 建议：<建议从哪里切入>
   - 建议：<建议先确认什么 / 比较什么 / 产出什么最小结果>
2. <第二个焦点>
   - 建议：<对应建议>
```

建议方案应满足：
- 足够具体，能让用户在另一台电脑或下一次会话中直接接着做
- 优先写成每条焦点下面的 1–3 条可执行建议，而不是泛泛口号
- 如果存在明显优先顺序，明确先后顺序
- 如果已经接近可执行阶段，直接给出下一步最小动作

示例：

```md
## 下一步聚焦
1. 先明确第一版是否只服务内部团队。
   - 建议：先对齐“内部使用”与“对外开放”两种路径的取舍标准。
   - 建议：如果目标是尽快验证流程闭环，优先按内部团队场景收窄范围。
   - 建议：等内部路径清楚后，再回头评估对外开放的必要性。
```

## Exit Criteria

满足下面任一条件时，主动建议结束本轮讨论或切换阶段：
- 当前真相已经足够支持实现、写作、拆解任务或正式成稿。
- 剩余问题已收缩为少量明确未决项，继续泛聊收益不高。
- 当前主题已经完成，适合转入新的更聚焦讨论。

常见下一阶段包括：
- 转成正式文档。
- 转成实现规划。
- 转成任务拆解。
- 单独开一轮讨论某个最高优先未决问题。

## Working Heuristics

优先遵守下面几条简单启发式：

1. **先收敛问题，再扩写答案。**
2. **先维护当前真相，再保留少量必要的不确定性。**
3. **先推进最有杠杆的问题，再处理细枝末节。**
4. **先帮用户看见“已经确定了什么”，再讨论“还差什么”。**
5. **当讨论已经够清楚时，主动结束讨论模式。**
6. **只要写了下一步，就把建议推进方案直接附在该条目下面。**
