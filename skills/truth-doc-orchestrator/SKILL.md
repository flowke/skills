# Truth Doc Orchestrator

将 `docs-specs-flo/` 重构为一套 **truth-driven documentation system**：先建立事实真相，再定义意图真相，再展开行为语义、工程落地与验证闭环，最终把结果沉淀为模块当前真相与共享真相。

## Philosophy

本 skill 遵循三条总原则：

1. **上下文即能力**  
   能力不来自静态模板堆积，而来自当前任务上下文中的事实、约束、目标、状态与已沉淀真相的正确装配。上下文缺口不同，工作模式与输出结构也应不同。
2. **真相即实现**  
   文档中的事实真相、意图真相、行为语义与当前真相，不是实现的附属说明，而是实现的正式上游。代码、配置、测试、运行约束与操作流程，都是这些真相的派生物。
3. **漂移即信号，不是中断**  
   发现真相与实现不一致时，优先显式记录、分级、隔离影响并指定下一步，而不是直接让推进失效。禁止静默漂移，允许受控推进。

## Core Truth Model

先区分 8 类信息：

- **事实来源**：原始依据，回答“凭什么这么说”
- **事实真相**：基于来源确认后的现实结论，回答“现在真实是什么”
- **意图真相**：本次希望系统成立的目标定义，回答“这次要变成什么样”
- **当前真相**：当前时点团队应优先信任的模块事实与能力总览
- **推断**：基于已知事实做出的合理分析，不等于已确认事实
- **假设**：为了推进而暂时接受的前提
- **待确认项**：仍未决定、会影响事实判断或意图定义的关键问题
- **漂移**：既有真相与当前实现或运行现实不一致的状态

## Repository Structure

默认文档根目录为 `docs-specs-flo/`。

- 全局导航：`index/00-全局导航.md`
- 共享真相：`shared-truths/`
- 模块入口：`modules/<模块名>/00-模块导航.md`
- 模块事实区：`modules/<模块名>/facts/`
- 活跃意图区：`modules/<模块名>/intents/active/<意图ID>-<意图名>/`
- 已沉淀意图区：`modules/<模块名>/intents/settled/`

模块事实区固定入口：
- `facts/00-事实索引.md`
- `facts/10-当前能力地图.md`

单个活跃意图固定使用以下 6 份主文档：
1. `00-意图总览.md`
2. `10-事实基础.md`
3. `20-意图图谱.md`
4. `30-行为语义.md`
5. `40-工程落地.md`
6. `50-验证与回归.md`

辅助目录：
- `attachments/`：补充图片、导出物、样例材料等

## Quick Start

面对一个新任务时，先不要急着进入固定模板，先按下面顺序判断：

1. **先判定当前上下文缺什么**
   - 缺来源：先补事实来源
   - 缺事实：先整理 `facts/`
   - 缺目标：先写 `00-意图总览.md`
   - 缺行为：先写 `30-行为语义.md`
   - 缺落地路径：先写 `40-工程落地.md`
   - 缺验证闭环：先写 `50-验证与回归.md`
   - 出现不一致：进入漂移校准

2. **若任务是正常推进**
   - 先补齐事实来源
   - 再确认事实真相
   - 再定义意图真相
   - 再展开行为、工程、验证
   - 最后更新当前真相并沉淀

3. **若任务中发现漂移**
   - 先判断这是“未知”还是“漂移”
   - 若是漂移，优先记录到 `60-漂移校准.md`
   - 完成等级判断、影响范围、受信切片、处理路径、下一步动作
   - 若可继续推进，再写 `61-带漂移推进.md`

4. **何时创建 `60-漂移校准.md`**
   - 既有真相与实现或运行现实不一致时
   - 下游已经可能被旧结论误导时
   - 需要明确“先修什么、还能做什么”时

5. **何时创建 `61-带漂移推进.md`**
   - 已识别漂移，但存在受信切片，且任务不能整体停掉时
   - 需要明确本轮允许推进范围、禁止误判范围与回收条件时

6. **何时可以继续推进**
   只有同时满足以下条件时，才允许带漂移推进：
   - 漂移已显式记录
   - 已完成分级
   - 已划定受信切片
   - 已写出临时假设
   - 已指定下一步校准动作

7. **何时才能进入沉淀**
   - 验证闭环完成
   - 假设已证实、证伪或转入待确认项
   - 漂移已回收、降级处理或显式转入当前真相
   - `facts/10-当前能力地图.md` 已更新
   - 模块导航与沉淀索引已更新

如果不确定当前属于哪一步，默认先做：
- `10-事实基础.md` 的事实与约束整理
- 或 `60-漂移校准.md` 的失配记录

优先把“现在真实是什么、哪里不一致、接下来做什么”写清楚，再继续推进。

## Working Modes

先识别当前上下文缺失的是来源、事实、意图、行为、工程、验证、沉淀还是漂移校准，再决定主工作模式。

1. 事实来源整理
2. 事实区维护
3. 活跃意图定义
4. 意图图谱整理
5. 行为语义整理
6. 工程落地规划
7. 验证与回归整理
8. 漂移校准与受控推进
9. 沉淀整理
10. 共享真相晋升
11. 模块导航维护

## Core Workflow

### A. 活跃意图标准顺序

1. 识别并补齐事实来源
2. 沉淀事实真相到 `facts/`
3. 写 `00-意图总览.md`
4. 写 `10-事实基础.md`
5. 写 `20-意图图谱.md`
6. 写 `30-行为语义.md`
7. 写 `40-工程落地.md`
8. 写 `50-验证与回归.md`
9. 意图完成后执行沉淀整理
10. 生成 `intents/settled/<意图ID>-<意图名>.md`
11. 更新 `facts/10-当前能力地图.md`
12. 必要时晋升到 `shared-truths/`
13. 更新 `00-模块导航.md`
14. 从 `intents/active/` 移除原完整目录

### B. 漂移出现时的标准顺序

1. 记录失配现象与事实来源
2. 判断漂移等级与类型
3. 划定影响范围与当前受信切片
4. 选择处理路径：修文档 / 修实现 / 临时并存 / 拆分意图
5. 写出临时假设、限制条件与下一步校准动作
6. 在受信切片内继续推进可继续的部分
7. 在验证或沉淀阶段回收漂移结果并更新当前真相

### C. 沉淀整理目标

沉淀的对象不是“过程文档本身”，而是：

- 新确认的事实真相
- 已兑现的意图真相
- 已证实或证伪的假设
- 已记录并处理的漂移
- 模块当前真相的更新
- 可跨模块复用的共享真相

## Document Semantics

- `facts/00-事实索引.md`：事实导航层
- `facts/10-当前能力地图.md`：当前真相总览层
- `00-意图总览.md`：意图真相入口层
- `10-事实基础.md`：意图成立的事实底座
- `20-意图图谱.md`：意图的正式图谱表达层
- `30-行为语义.md`：意图到系统行为的映射层
- `40-工程落地.md`：受事实约束的工程实施层
- `50-验证与回归.md`：意图兑现验证层
- `intents/settled/<意图>.md`：真相更新结果层

## Robustness Rules

- 漂移是信号，不是默认中断
- 禁止静默漂移
- 允许显式漂移下的受控推进
- Gate 约束的是完成声明，不是所有局部动作
- 只要漂移已记录、已分级、已隔离影响、已指定下一步，就允许在受信切片内继续

### 漂移分级

- `D0`：观察到差异，但不影响当前推进
- `D1`：局部漂移，可隔离影响
- `D2`：关键漂移，影响当前意图成立
- `D3`：真相基座失效，需要优先重建

### 受控推进最小条件

满足以下条件时，可在漂移存在时继续推进：

1. 漂移不是静默存在
2. 影响范围已明确
3. 当前受信切片已明确
4. 临时假设已写出
5. 后续验证或校准动作已指定

## Hard Gates

- **事实未立，原则上不进入意图；若已识别漂移范围且可隔离影响，可在显式假设下进入受控意图定义**
- **意图未清，原则上不进入行为；若仅局部目标存在漂移，可对已稳定子目标继续展开行为语义**
- **行为未清，原则上不进入工程；若存在已确认的稳定行为切片，可在受信切片内继续工程落地**
- **验证未闭环，不算完成，但不等于不能继续做事实整理、漂移校准或局部实现**
- **当前真相未更新，不算沉淀完成**

## Mermaid Usage Rule

Mermaid 是正式表达工具，不是装饰。

当出现以下情况时，优先用 Mermaid：
- 角色多于 2 个
- 状态多于 3 个
- 存在异步交互
- 存在明显分支、回退、补偿逻辑
- 纯文字已经难以稳定表达意图或行为

推荐图种：
- `flowchart`：目标路径、模块结构、分支流转
- `sequenceDiagram`：角色交互、接口时序
- `stateDiagram-v2`：状态机、生命周期
- `erDiagram`：对象关系、领域结构
- `journey`：用户路径

## Output Contracts

### 漂移校准固定输出结构

执行“漂移校准与受控推进”时，优先输出为：

```md
# <模块名> <对象名> 漂移校准结果

## 1. 观察到的漂移
## 2. 漂移等级与类型
## 3. 影响范围
## 4. 当前受信切片
## 5. 临时假设与限制
## 6. 处理路径
## 7. 下一步校准动作
```

若某项为空，显式写“无”或“本次无需更新”。

### 沉淀结果固定输出结构

执行“沉淀整理”时，优先输出为：

```md
# <模块名> <意图ID>-<意图名> 沉淀结果

## 1. 新确认的事实真相
## 2. 已兑现的意图真相
## 3. 假设验证结果
## 4. 漂移处理结果
## 5. 当前真相更新
## 6. 共享真相晋升
## 7. 原活跃意图目录处理
```

若某项为空，显式写“无”或“本次无需更新”。
