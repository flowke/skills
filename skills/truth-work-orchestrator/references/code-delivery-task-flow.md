# Code Delivery Task Flow

当 `truth-work-orchestrator` 需要创建、维护或收口一个 **代码落地 task** 时，优先参考这里。

目标：
- 让代码落地 task 从“只是一个任务目录”升级为“有明确推进骨架的执行对象”
- 保持 task 文档轻量，但足够支撑对齐、规划、执行承接与回写
- 明确代码落地 task 与回归测试 task 的边界，避免重新回到旧的 `planning.md` / `regression.md` 阶段文件模型

## 1. 核心目标

当前这类 task 的默认优化目标是三件事同时成立：
- **最优落地路径**：优先选择最符合 truth、依赖最短、改动面最可控、最容易验证的实施路径
- **最小返工**：尽量在进入编码前把范围、边界、依赖与顺序收敛清楚，避免边做边推翻
- **可中断重启**：任何一次中断后，都应能仅通过 task 目录中的文档恢复当前进度，而不依赖聊天上下文或短期记忆

## 2. 边界与定位

代码落地 task 用于承接：
- 新功能实现
- 既有逻辑改造
- 缺陷修复
- 围绕模块 / 主题 / 子模块发起的一次代码级落地活动

它不负责：
- 代替稳定对象 truth
- 变成冗长的过程日志

默认理解：
- **模块 / 主题 / 子模块** 负责维护稳定 truth
- **代码落地 task** 负责承接一次实现活动
- **代码落地 task** 默认覆盖实现、验证、必要回归与结果收口
- **回归测试 task** 仅在验证目标明显独立、范围明显外扩时按需使用

## 3. 标准流程骨架

### 3.1 实施路径选择原则

在进入具体实现前，优先比较候选路径，默认优先选择同时满足下面条件的路径：
- **truth 对齐成本最低**：最不容易偏离当前稳定 truth
- **改动面最小**：优先修改必要的最小范围，而不是一开始就大面积扩散
- **依赖链最短**：优先避免引入不必要的上下游联动
- **验证路径最短**：更容易快速确认“这条路是对的”
- **回滚成本更低**：如果判断失误，更容易撤回或改向
- **文档续推更容易**：另一台电脑或另一轮会话中更容易接上

如果存在两条都能做通的路径，默认优先：
- 更少文件 / 更少模块联动的路径
- 更少隐含假设的路径
- 更容易阶段性交付和验证的路径


### 阶段 1：创建与挂接

目标：让 task 先成为一个明确对象。

此阶段至少要明确：
- 任务类型：`代码落地`
- 挂接对象：模块 / 主题 / 子模块，或暂时为空
- 当前状态：建议从 `待对齐` 或 `对齐中` 起步
- 当前目标：这次代码落地到底要解决什么

默认动作：
- 创建 task 目录
- 建立 `task.md`
- 如果 task 来自 intake，可补 `来源 intake 项`

### 阶段 2：任务对齐

目标：把“准备做代码”收敛成“做什么、不做什么、依据是什么”。

至少覆盖：
- 本轮目标
- 本轮范围 / 非目标
- 关键约束
- 输入依据
- 挂接对象是否正确

说明：
- 如果当前信息明显不足，先补资料，不急着进入规划
- 对齐结果优先沉淀回 `task.md`，而不是先新开更多文件

### 阶段 3：一致性复盘

目标：在进入实现规划前，显式检查任务是否已经“足够稳”。

至少检查：
- 是否与当前 truth 冲突
- 是否影响其他模块 / 主题 / 子模块
- 是否遗漏关键范围或依赖
- 是否还有未决决策会阻塞执行

说明：
- 这是代码落地 task 进入规划前的必要环节
- 可以轻量，但不能跳过
- 如果发现当前 truth 本身不稳，应先回到稳定对象层修正，再继续推进 task

### 阶段 4：实现规划

目标：把代码落地任务转成可执行的工作块，并明确为什么选这条落地路径。

此阶段建议明确：
- 候选路径对比（如存在多个方案）
- 最终采用路径
- 选择理由
- 实现拆解
- 执行顺序
- 风险与依赖
- 是否需要拆子任务
- 输出去向

文档策略：
- 简单任务：直接写在 `task.md`
- 稍复杂任务：新增 `implementation-plan.md`

说明：
- `implementation-plan.md` 是 **task 目录内** 的规划文档
- 不要重新把模块 / 主题层的 `planning.md` 当作默认结构复活

### 阶段 5：执行承接

目标：把规划好的 task 交给实际执行层。

可能的承接方式：
- 直接进入实现
- 拆分为多个 `subtask-*.md`
- 生成明确交接说明给执行 agent / 执行人

文档策略：
- 需要轻量拆分时，使用 `subtask-*.md`
- 需要显式交接说明时，使用 `handoff.md`

### 阶段 6：执行中回填

目标：避免执行发现的新事实只留在代码里，不回到 task / truth。

执行过程中如果出现下面情况，应回填：
- 原理解不完整
- 范围需要收缩或扩张
- 依赖关系与预期不一致
- 发现稳定对象 truth 过时

默认动作：
- 先更新 `task.md` 的当前状态 / 当前阻塞 / 下一步动作
- 如果影响稳定对象边界，再回写对应 `current-truth.md` 或相关 knowledge

### 阶段 7：验证与回归

目标：在代码实施完成后，直接在同一个 task 内完成必要验证与回归。

此阶段至少要明确：
- 验证范围
- 验证方式
- 验证结果
- 是否发现偏差 / 缺陷
- 是否需要回写 truth 或继续修正代码

说明：
- 默认不为这类验证再新起一个 task
- 代码落地 task 应直接承接“实现 → 验证 → 修正 → 再验证”的闭环
- 只有在验证目标明显独立、范围明显外扩时，才按需考虑单独的回归测试 task

### 阶段 8：完成收口与回写

目标：让 task 的完成结果可追溯、可交接、可沉淀，并完成最终回写。

至少收口：
- 本次完成了什么
- 验证 / 回归结论是什么
- 哪些未完成 / 明确不做
- 产出落到了哪里
- 是否还有后续待跟进事项

默认回写方向：
- 模块 / 主题 / 子模块 truth
- 模块内或顶层 knowledge

边界说明：
- **代码落地 task 默认自己闭环，不再默认外拆验证 / 回归 task**
- 如果验证中发现 truth 过时，应更新 truth
- 如果验证中发现 code 偏离，应继续在当前 task 中修正并完成闭环

## 4. 推荐状态集合

当前建议代码落地 task 使用下面这些状态：
- `待对齐`
- `对齐中`
- `待规划`
- `规划中`
- `待执行`
- `执行中`
- `待回写`
- `已完成`
- `已挂起`

说明：
- 不要求所有 task 都严格走完每一个状态
- 但尽量避免只用“进行中 / 已完成”两种过粗状态
- 建议把验证 / 回归纳入同一个状态机中，而不是默认外拆成新 task

## 5. 代码落地 task 的推荐文档集合

### 强必有
- `task.md`

### 按需添加
- `implementation-plan.md`
- `subtask-*.md`
- `handoff.md`
- `verification.md`
- `attachments/`

### 各文档职责

#### `task.md`
作为 task 的主入口与最小真相文档，用来维护：
- 任务类型
- 挂接对象
- 当前状态
- 当前目标
- 输入依据
- 下一步动作
- 输出去向
- 验证与回归结论

#### `implementation-plan.md`
用于承接较复杂任务的实现规划，推荐在下面情况创建：
- 需要明确实现拆解
- 有多步执行顺序
- 存在明显风险 / 依赖
- 需要对子任务进行统一编排

#### `subtask-*.md`
用于轻量拆分父 task，不单独升级为目录。

#### `handoff.md`
用于显式交接，推荐在下面情况创建：
- 任务需要切到执行 agent / 执行人
- 任务需要交给另一个人继续推进
- 任务在中断后需要低成本恢复

#### `verification.md`
用于承接较复杂任务的验证 / 回归记录，推荐在下面情况创建：
- 验证范围较大
- 存在多轮修正与复测
- 需要把验证证据、结果与结论独立记录

#### `attachments/`
用于承接 task 私有的资料、截图、导出物、表格等，不适合直接混在正文里时使用。

## 6. 推荐目录形态

### 最小形态

```text
YYMMDD-<中文任务名>/
├── task.md
```

### 常见扩展形态

```text
YYMMDD-<中文任务名>/
├── task.md
├── implementation-plan.md
├── handoff.md
├── verification.md
├── subtask-ui.md
├── subtask-data.md
└── attachments/
```

## 7. 可中断重启规则

优先遵守下面规则：

1. `task.md` 必须始终能回答四个问题：**现在在做什么、为什么这么做、已经做到哪里、下一步做什么**。
2. 每当实施路径、范围、顺序、阻塞发生变化时，优先先更新文档，再继续推进实现。
3. 小任务至少回填 `task.md` 的 `当前状态`、`当前进展`、`下一步动作`。
4. 中大型任务在准备切换设备、切换会话或暂停时，建议补 `handoff.md`，明确“从哪里继续”。
5. 文档应默认服务于“另一台电脑上的自己”也能接上；不要把关键上下文只留在聊天里。

## 8. 回写规则

优先遵守下面规则：

1. `task.md` 写执行对象的当前状态，不替代稳定对象 truth。
2. 任务过程中出现的稳定结论，应回写到对应模块 / 主题 / 子模块，而不是长期滞留在 task 内。
3. 共享知识应进入 `knowledge/` 或模块内 `knowledge/`，不要把可复用知识长期堆在 task 目录里。
4. 如果实现发现 truth 不稳，应先修 truth，再继续推进 task。
5. 如果实现已经完成但仍需验证是否符合 truth，默认继续在当前代码落地 task 中完成验证与回归闭环；不要机械外拆成新 task。

## 9. 轻量判断规则

遇到下面情况时，直接只维护 `task.md` 即可：
- 改动范围小
- 依赖少
- 不需要拆子任务
- 规划可以在几条 bullet 内说清

遇到下面情况时，建议补 `implementation-plan.md`：
- 需要多阶段实施
- 需要先后顺序与风险控制
- 同时涉及多个文件区块 / 多个子模块
- 任务会在多人或多轮之间交接

遇到下面情况时，建议补 `handoff.md`：
- 执行交接成本高
- 中断后需要快速恢复
- 只看 `task.md` 难以直接接上工作

遇到下面情况时，建议补 `verification.md`：
- 验证 / 回归步骤明显复杂
- 存在多轮修正与复测
- 需要把验证证据与最终结论分开记录
