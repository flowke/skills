# Object Templates

当 `truth-work-orchestrator` 需要创建或维护 `intake / task / module / topic / knowledge` 等对象时，优先参考这里的**轻量模板建议**。

注意：
- 这里提供的是**推荐结构**，不是强制 schema。
- 目标是帮助产物保持可读、可继续推进、可回写，而不是把系统做成厚重的管理模板。
- 如果对象已经有稳定结构，优先延续现有结构，而不是为了模板强行重写。

## General Rules

- 稳定对象优先写清“当前已经成立什么”，不要把执行细节压进 `current-truth.md`。
- 执行对象优先写清“现在要做什么、挂在哪、状态如何”，不要把 task 写成冗长日志。
- 共享知识优先写清“这是什么、适用于哪里”，不要强制统一所有知识文档格式。
- 模板应默认轻量；只有当场景明确需要时，再补更重章节。
- 代码落地 task 的流程与文档关系，优先结合 `references/code-delivery-task-flow.md` 一起使用。
- 独立回归任务的流程与文档关系，优先结合 `references/regression-task-flow.md` 一起使用。
- 可以使用 checkbox 帮助维护规划或执行进度，但 checkbox 只表示某个动作 / 检查点是否完成，**不能替代** `当前状态`、`当前进展`、`验证与回归结论` 这些主语义字段。
- 对代码落地 task 而言，推荐把 checkbox 放在 `当前实现计划`、`下一步动作`、`implementation-plan.md` 或 `subtask-*.md` 中，而不是只靠勾选数量判断 task 是否完成。

## 1. intake / note.md

适用场景：
- 资料先入队，尚未正式创建 task

补充说明：
- 如果当前内容属于“待归类记忆”，也可以先复用这一对象承接

### 推荐最小字段
- `原始描述`
- `来源`
- `当前状态`
- `备注`

### 可选字段
- `候选挂接对象`
- `是否已转 task`
- `相关链接`

### Minimal Template

```md
# Intake Note

- 原始描述：
- 来源：
- 当前状态：待处理
- 备注：

## 相关资料
- 
```

## 4. tasks / implementation-plan.md

适用场景：
- 代码落地 task 需要单独承接实现规划
- 任务比几条 bullet 更复杂

### 推荐最小字段
- `规划目标`
- `输入依据`
- `规划检查清单`
- `实现拆解`
- `执行顺序 / 优先级`

### 可选字段
- `风险与依赖`
- `待确认事项`
- `当前结论`

### Minimal Template

```md
# Implementation Plan

## 规划目标
- 本轮目标：
- 规划产出：

## 输入依据
```

## 9. modules / topics / current-truth.md

适用场景：
- 记录稳定对象的当前真相

### 推荐最小结构
- `讨论主题`
- `当前真相`
- `待确认项`
- `未决问题`
- `下一步聚焦`

### Notes
- 模块与主题共用同一 truth 骨架即可。
- 模块如果存在局部职责拆分，默认使用轻量子模块 `submodule-*.md`。
- 只有在当前上下文被显式说明为 **目录子模块** 时，才使用 `submodule-*/current-truth.md`。
- 目录子模块内部可继续包含 `knowledge/` 与下一层 `submodule-*`，形成分形结构。
- 不要把 task 执行流水直接堆进 `current-truth.md`。

- 模块内部需要独立维护一个子模块 truth

### 推荐最小字段
- `子模块职责`
- `当前真相`

```md
- 

## 待确认项
- 

## 未决问题
-
```

## 10.1 modules / submodule-*/（目录子模块，支持分形）

适用场景：
- 当前已被显式说明为 **目录子模块**
- 子模块已经需要独立知识层
- 子模块需要继续拆下一层子模块
- 子模块会长期演化，单个 `submodule-*.md` 已经承载不下

### 推荐最小结构
- `current-truth.md`
- `knowledge/`（按需）
- 下一层 `submodule-*`（按需，可为文件或目录）

### 推荐目录形态

```text
submodule-xxx/
├── current-truth.md
├── knowledge/
├── submodule-aaa.md
└── submodule-bbb/
    ├── current-truth.md
    └── knowledge/
```

### Notes
- 目录子模块的主入口固定为 `current-truth.md`。
- 目录子模块目录内部继续遵守与模块类似的规则：`current-truth.md` 放结论，`knowledge/` 放知识，下一层 `submodule-*` 按需出现。
- 默认优先保持轻量；只有被显式说明为 **目录子模块** 且复杂度真的外扩时，才把轻量子模块升级为目录结构。

## 11. knowledge /

适用场景：
- 记录可跨对象复用的协议、机制、约束、长期约定等知识

### 当前策略
- 不强制规定固定正文模板
- 允许按知识内容自由组织
- 推荐补充轻量元信息以便追溯与复用，但不把这些字段升级为必填 schema

### 推荐可选元信息（默认不强制）
- `来源`
- `更新时间`
- `适用范围`
- `相关对象`

```md
- 相关对象：

```

## 12. Drift Handling Notes

适用场景：
- 回归测试 task 发现 code 与 truth 不一致

### 推荐记录字段
- `漂移分类`
- `当前判断依据`
- `默认动作`
- `回写位置`

### 分类建议
- `truth 过时`
- `code 偏离`
- `暂时无法判断`

### Notes
- 先分类，再动作。
- 证据不足时，不自动改 truth，也不自动修代码。
- 如果需要把冲突升级回稳定对象层，应在对应模块 / 主题的 `待确认项` 或 `未决问题` 中留痕。
