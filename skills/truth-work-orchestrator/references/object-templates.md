# Object Templates

当 `truth-work-orchestrator` 需要创建或维护 `intake / task / module / topic / knowledge` 等对象时，优先参考这里的**轻量模板建议**。

注意：
- 这里提供的是**推荐结构**，不是强制 schema。
- 目标是帮助产物保持可读、可继续推进、可回写，而不是把系统做成厚重的管理模板。
- 如果对象已经有稳定结构，优先延续现有结构，而不是为了模板强行重写。

## General Rules

- 稳定对象优先写清“当前已经成立什么”，不要把执行细节压进 `current-truth.md`。
- 执行对象优先写清“现在要做什么、挂在哪、状态如何”，不要把 task 写成冗长日志。
- 共享知识优先写清“这是什么、适用于哪里”，不要强制统一所有知识文档格式。
- 模板应默认轻量；只有当场景明确需要时，再补更重章节。
- 代码落地 task 的流程与文档关系，优先结合 `references/code-delivery-task-flow.md` 一起使用。

## 1. intake / note.md

适用场景：
- 资料先入队，尚未正式创建 task

### 推荐最小字段
- `原始描述`
- `来源`
- `当前状态`
- `备注`

### 可选字段
- `候选挂接对象`
- `是否已转 task`
- `相关链接`

### Minimal Template

```md
# Intake Note

- 原始描述：
- 来源：
- 当前状态：待处理
- 备注：

## 相关资料
- 
```

- `当前阻塞`
- `相关子任务`

- 挂接对象：
- 当前状态：

## 当前目标
- 

## 输入依据
-

- 
```

## 3. 代码落地 task / task.md（推荐扩展骨架）

适用场景：
- 任务类型已确定为 `代码落地`
- 需要把对齐、规划、执行承接与回写维护在同一个 task 主入口中

### 推荐字段
- `任务类型：代码落地`
- `挂接对象`
- `当前状态`
- `来源 intake 项`（可选）
- `当前阻塞`（可选）

## 4. tasks / implementation-plan.md

适用场景：
- 代码落地 task 需要单独承接实现规划
- 任务比几条 bullet 更复杂

### 推荐最小字段
- `规划目标`
- `输入依据`
- `实现拆解`
- `执行顺序 / 优先级`

### 可选字段
- `风险与依赖`
- `待确认事项`

## 规划目标
- 本轮目标：
- 规划产出：

## 输入依据
- 依据文档：
- 关键前提：

  - 产出：
  - 注意点：

- 后做什么：
- 可并行项：

- 

## 待确认事项
- 
```

## 8. modules / topics / current-truth.md

适用场景：
- 记录稳定对象的当前真相

### 推荐最小结构
- `讨论主题`
- `当前真相`
- `待确认项`
- `未决问题`
- `下一步聚焦`

### Notes
- 模块与主题共用同一 truth 骨架即可。
- 模块如果存在局部职责拆分，可把局部真相写到 `submodule-*.md`。
- 不要把 task 执行流水直接堆进 `current-truth.md`。

## 10. knowledge /

适用场景：
- 记录可跨对象复用的协议、机制、约束、长期约定等知识

### 当前策略
- 不强制规定固定正文模板
- 允许按知识内容自由组织
- 推荐补充轻量元信息以便追溯与复用，但不把这些字段升级为必填 schema

### 推荐可选元信息（默认不强制）
- `来源`
- `更新时间`
- `适用范围`
- `相关对象`

### Light Example

```md
# <知识标题>

- 来源：
- 更新时间：
- 适用范围：
- 相关对象：

## 内容
- 
```

## 11. Drift Handling Notes

适用场景：
- 回归测试 task 发现 code 与 truth 不一致

### 推荐记录字段
- `漂移分类`
- `当前判断依据`
- `默认动作`
- `回写位置`

### 分类建议
- `truth 过时`
- `code 偏离`
- `暂时无法判断`

### Notes
- 先分类，再动作。
- 证据不足时，不自动改 truth，也不自动修代码。
- 如果需要把冲突升级回稳定对象层，应在对应模块 / 主题的 `待确认项` 或 `未决问题` 中留痕。
