# Delivery Work-item Flow

当 `truth-work-orchestrator` 需要创建、维护或收口一个 **delivery work-item** 时，优先参考这里。

> 注：这里的 **delivery work-item / 工作承接对象** 不应被理解为狭义执行动作。

目标：
- 让delivery work-item 从“只是一个工作项目录”升级为“有明确推进骨架的工作承接对象”
- 保持 work-item 文档轻量，但足够支撑对齐、规划、执行承接与回写
- 明确 delivery work-item 自身负责实施 / 验证 / 回写闭环，不再额外引入单独的检查类 work-item。
- 让当前 work-item 区始终保持可读：完成 work-item 退出 `work-items/`，进入顶层 `archive/work-items/`

## Module Demand → Work-item-first Rule

当用户在某个模块 / 子模块上提出**足够明确、明显会进入实施**的新需求或改需求时，默认推荐：
1. **立即创建挂接该对象的 work-item**，作为执行承接对象
2. **同时在模块 / 子模块 truth 中留一条极轻量的 `intent` / `declared` 痕迹**，避免模块对象失真
3. work-item 完成后，把稳定结果回补到挂接对象
4. **如果 work-item 挂接了模块 / 子模块 / 主题，则回补完成前不能标记为 `已完成`**
5. 已完成 work-item 不继续留在 `work-items/`，而是立即转入 `archive/work-items/`

## 1. 核心目标

当前这类 work-item 的默认优化目标是三件事同时成立：
- **最优落地路径**：优先选择最符合 truth、依赖最短、改动面最可控、最容易验证的实施路径
- **最小返工**：尽量在进入编码前把范围、边界、依赖与顺序收敛清楚，避免边做边推翻
- **可中断重启**：任何一次中断后，都应能仅通过 work-item 目录中的文档恢复当前进度，而不依赖聊天上下文或短期记忆

## 2. 边界与定位

delivery work-item 用于承接：
- 新功能实现
- 既有逻辑改造
- 缺陷修复
- 围绕模块 / 主题 / 子模块发起的一次交付型工作推进

它不负责：
- 代替稳定对象 truth
- 变成冗长的过程日志

默认理解：
- **模块 / 主题 / 子模块** 负责维护稳定 truth
- **delivery work-item** 负责承接一次交付型工作推进，而不是只承接执行动作
- **delivery work-item** 默认覆盖对齐、规划、实现、验证、必要检查与结果收口
- 默认不要外拆独立的检查类 work-item；验证与检查优先在当前 delivery work-item 内闭环完成。

## 3. 标准流程骨架

### 3.1 实施路径选择原则

在进入具体实现前，优先比较候选路径，默认优先选择同时满足下面条件的路径：
- **truth 对齐成本最低**：最不容易偏离当前稳定 truth
- **改动面最小**：优先修改必要的最小范围，而不是一开始就大面积扩散
- **依赖链最短**：优先避免引入不必要的上下游联动
- **验证路径最短**：更容易快速确认“这条路是对的”
- **回滚成本更低**：如果判断失误，更容易撤回或改向
- **文档续推更容易**：另一台电脑或另一轮会话中更容易接上

如果存在两条都能做通的路径，默认优先：
- 更少文件 / 更少模块联动的路径
- 更少隐含假设的路径
- 更容易阶段性交付和验证的路径


### 3.2 Checkbox 使用规则

delivery work-item 允许并推荐使用 checkbox 来表达规划或执行中的检查点，但默认遵守下面边界：
- checkbox 主要用于表达“某个规划动作 / 执行动作 / 验证动作是否完成”
- checkbox **不能替代** `当前状态`、`当前进展`、`验证与检查结论` 这些主语义字段
- 勾掉某个执行项，不等于整个 work-item 已完成；只有在状态、验证和回写都更新后，work-item 才能视为真正收口
- 优先勾掉“有状态意义的检查点”，例如完成主逻辑修改、补测试、完成最小验证、完成回写；不要把 checkbox 细化成过多碎动作，制造假进度
- 如果 work-item 较复杂，推荐：`implementation-plan.md` 维护规划 checklist，`work.md` 维护执行 checklist，`sub-*.md` 维护子工作项 checklist

### 阶段 1：创建与挂接

目标：让 work-item 先成为一个明确对象。

此阶段至少要明确：
- 工作项类型：`delivery`
- 挂接对象：模块 / 主题 / 子模块，或暂时为空
- 当前状态：建议从 `待对齐` 或 `对齐中` 起步
- 当前目标：这次delivery要承接哪一段交付型工作推进

默认动作：
- 在 `docs-TWO/work-items/` 下创建 work-item 目录
- 建立 `work.md`
- 如果 work-item 来自 intake，可补 `来源 intake 项`
- 如果 work-item 挂接到模块 / 子模块 / 主题，默认同步在对应 truth 中留一条极轻量的 `intent` / `declared` 痕迹

### 阶段 2：工作项对齐

目标：把“准备做代码”收敛成“做什么、不做什么、依据是什么”。

至少覆盖：
- 本轮目标
- 本轮范围 / 非目标
- 关键约束
- 输入依据
- 挂接对象是否正确

说明：
- 如果当前信息明显不足，先补资料，不急着进入规划
- 对齐结果优先沉淀回 `work.md`，而不是先新开更多文件

### 阶段 3：一致性复盘

目标：在进入实现规划前，显式检查任务是否已经“足够稳”。

至少检查：
- 是否与当前 truth 冲突
- 是否影响其他模块 / 主题 / 子模块
- 是否遗漏关键范围或依赖
- 是否还有未决决策会阻塞执行

说明：
- 这是delivery work-item 进入规划前的必要环节
- 可以轻量，但不能跳过
- 如果发现当前 truth 本身不稳，应先回到稳定对象层修正，再继续推进该 work-item

### 阶段 4：实现规划

目标：把delivery work-item转成可执行的工作块，并明确为什么选这条落地路径。

此阶段建议明确：
- 候选路径对比（如存在多个方案）
- 最终采用路径
- 选择理由
- 实现拆解
- 执行顺序
- 风险与依赖
- 是否需要拆子工作项
- 输出去向

文档策略：
- 简单任务：直接写在 `work.md`
- 稍复杂任务：新增 `implementation-plan.md`

说明：
- `implementation-plan.md` 是 **work-item 目录内** 的规划文档
- 不要重新把模块 / 主题层的 `planning.md` 当作默认结构复活

### 阶段 5：执行承接

目标：把规划好的 work-item 交给实际执行层，同时保持主入口文档随时可续推。

可能的承接方式：
- 直接进入实现
- 拆分为多个 `sub-*.md`
- 把关键承接信息直接写回 `work.md` / `sub-*.md` 主入口
- 仅在主入口文档不足以承载复杂交接上下文时，再补 `handoff.md`

文档策略：
- 需要轻量拆分时，使用 `sub-*.md`
- 默认优先更新 `work.md` / `sub-*.md`，让它们自己可接力
- 只有复杂交接说明难以内嵌进主入口时，才使用 `handoff.md`

### 阶段 6：执行中回填

目标：避免执行发现的新事实只留在代码里，不回到 work-item / truth。

执行过程中如果出现下面情况，应回填：
- 原理解不完整
- 范围需要收缩或扩张
- 依赖关系与预期不一致
- 发现稳定对象 truth 过时

默认动作：
- 先更新 `work.md` 的当前状态 / 当前阻塞 / 下一步动作
- 如果影响稳定对象边界，再回写对应 `current-truth.md` 或相关 knowledge

### 阶段 7：验证与检查

目标：在代码实施完成后，直接在同一个 work-item 内完成必要验证与检查。

此阶段至少要明确：
- 验证范围
- 验证方式
- 验证结果
- 是否发现偏差 / 缺陷
- 是否需要回写 truth 或继续修正代码

说明：
- 默认不为这类验证再新起一个 work-item
- delivery work-item 应直接承接“实现 → 验证 → 修正 → 再验证”的闭环
- 默认在当前 work-item 内完成验证与检查；不要机械外拆新的 work-item。

### 阶段 8：完成收口与回写

目标：让 work-item 的完成结果可追溯、可续推、可沉淀，并完成最终回写。

至少收口：
- 本次完成了什么
- 验证 / 检查结论是什么
- 哪些未完成 / 明确不做
- 产出落到了哪里
- 是否还有后续待跟进事项

默认回写方向：
- 模块 / 主题 / 子模块 truth
- 模块内或顶层 knowledge

边界说明：
- **delivery work-item 默认自己闭环，不再默认外拆验证 / 检查对象**
- 如果验证中发现 truth 过时，应更新 truth
- 如果验证中发现 code 偏离，应继续在当前 work-item 中修正并完成闭环

### 阶段 8.1 完成定义与归档

默认遵守下面规则：
- **挂接了模块 / 子模块 / 主题的 work-item，只有在回补挂接对象完成后，才可标记为 `已完成`**。
- 如果代码与验证已经收口，但挂接对象尚未更新，正确状态应是 `待回写`，而不是 `已完成`。
- `已完成` 是 work-item 的语义终态，但不应长期停留在 `work-items/`。
- work-item 一旦标记 `已完成`，应尽快从 `docs-TWO/work-items/` 移动到 `docs-TWO/archive/work-items/`。
- archive 表示退出当前工作区，不表示删除；默认保留 work-item 以支持追溯、恢复和证据链查看。


## 4. 推荐状态集合

当前建议delivery work-item 使用下面这些状态：
- `待对齐`
- `对齐中`
- `待规划`
- `规划中`
- `待执行`
- `执行中`
- `待回写`
- `已完成`
- `已挂起`

说明：
- 不要求所有 work-item 都严格走完每一个状态
- `待回写` 是一个重要收口状态：对挂接对象的 work-item 来说，回补完成前不应跳到 `已完成`
- `已完成` 是终态，默认应很快转入 `archive/work-items/`，而不是长期停留在 `work-items/`
- 建议把验证 / 检查纳入同一个状态机中，而不是默认外拆成新的 work-item

## 5. delivery work-item 的推荐文档集合

### 强必有
- `work.md`

### 按需添加
- `implementation-plan.md`
- `sub-*.md`
- `verification.md`
- `attachments/`
- `handoff.md`（仅在主入口文档不足以承载复杂交接上下文时）

### 各文档职责

#### `work.md`
作为 work-item 的主入口与最小真相文档，用来维护：
- 工作项类型
- 挂接对象
- 当前状态
- 当前目标
- 输入依据
- 当前实现计划（可使用 checkbox）
- 当前进展
- 接力入口 / 恢复入口
- 下一步动作（可使用 checkbox）
- 输出去向
- 验证与检查结论

#### `implementation-plan.md`
用于承接较复杂任务的实现规划，推荐在下面情况创建：
- 需要明确实现拆解
- 有多步执行顺序
- 存在明显风险 / 依赖
- 需要对子工作项进行统一编排

#### `sub-*.md`
用于轻量拆分父 work-item，不单独升级为目录；默认也应能独立表达当前状态、执行检查点、当前进展与接力入口。
- 在 `docs-TWO/work-items/<work-item>/` 下，`sub-*` 默认表示该父 work-item 的子工作项。

#### `handoff.md`
作为增强型交接说明，推荐只在下面情况创建：
- 主入口文档不足以承载复杂交接上下文
- 工作项需要跨多人 / 多轮长期接力，且额外压缩说明明显更省成本
- 需要把复杂背景、风险提醒或接手约束集中写给新的执行者

#### `verification.md`
用于承接较复杂任务的验证记录，推荐在下面情况创建：
- 验证范围较大
- 存在多轮修正与复测
- 需要把验证证据、结果与结论独立记录

#### `attachments/`
用于承接 work-item 私有的资料、截图、导出物、表格等，不适合直接混在正文里时使用。

## 7. 可中断重启规则

优先遵守下面规则：

1. `work.md` 必须始终能回答四个问题：**现在在做什么、为什么这么做、已经做到哪里、下一步做什么**。
2. `sub-*.md` 如果存在，也应默认能独立回答：**自己负责什么、做到哪里、接下来做什么**。
3. 每当实施路径、范围、顺序、阻塞发生变化时，优先先更新主入口文档，再继续推进实现。
4. 小任务至少回填 `work.md` 的 `当前状态`、`当前进展`、`接力入口`、`下一步动作`。
5. 中大型任务在准备切换设备、切换会话或暂停时，优先补全 `work.md` / `sub-*.md` 的接力信息；只有主入口明显承载不下时，才补 `handoff.md`。
6. work-item 一旦完成收口，应退出 `work-items/` 并转入 `archive/work-items/`，不要让已完成 work-item 混在当前工作区里。
7. 文档应默认服务于“另一台电脑上的自己”也能接上；不要把关键上下文只留在聊天里。

## 8. 回写规则

优先遵守下面规则：

1. `work.md` 写执行对象的当前状态，不替代稳定对象 truth。
2. 工作项过程中出现的稳定结论，应回写到对应模块 / 主题 / 子模块，而不是长期滞留在 work-item 内。
3. 共享知识应进入 `knowledge/` 或模块内 `knowledge/`，不要把可复用知识长期堆在 work-item 目录里。
4. 如果实现发现 truth 不稳，应先修 truth，再继续推进该 work-item。
5. 如果实现已经完成但仍需验证是否符合 truth，默认继续在当前 delivery work-item 中完成验证与检查闭环；不要机械外拆成新的 work-item。
6. 如果 work-item 挂接了模块 / 子模块 / 主题，则回补完成前不能标记为 `已完成`。

## 9. 轻量判断规则

遇到下面情况时，直接只维护 `work.md` 即可：
- 改动范围小
- 依赖少
- 不需要拆子工作项
- 规划可以在几条 bullet 内说清

遇到下面情况时，建议补 `implementation-plan.md`：
- 需要多阶段实施
- 需要先后顺序与风险控制
- 同时涉及多个文件区块 / 多个子模块
- 工作项会在多人或多轮之间续推

遇到下面情况时，建议补 `handoff.md`：
- 主入口文档难以承载复杂交接上下文
- 执行交接成本高，额外压缩说明明显更省成本
- 只看 `work.md` / `sub-*.md` 仍难以直接接上工作

遇到下面情况时，建议补 `verification.md`：
- 验证 / 检查步骤明显复杂
- 存在多轮修正与复测
- 需要把验证证据与最终结论分开记录
