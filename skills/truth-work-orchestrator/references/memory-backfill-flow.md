# Memory Backfill Flow

当 `truth-work-orchestrator` 需要把 **skill 外发生的代码修复 / 需求实现 / 技术判断** 回填成后续可复用上下文时，优先参考这里。

目标：
- 把“已经发生、但没有走 truth-work 流程”的变化，补回到当前产出体系中
- 让下次开发或修改功能时，大模型能先读到这些稳定上下文
- 避免代码已经变了，但 truth / knowledge 还停留在旧状态

## 1. 核心定位

这不是一个新的顶层对象类型，而是一条**回填流程**。

默认理解：
- 输入：skill 外发生的实现、修复、技术判断、踩坑经验、机制认知
- 动作：抽取其中值得长期保留的稳定信息
- 输出：回填到现有对象层，例如模块 truth、模块 knowledge、顶层 knowledge，或暂时进入 intake 待归类

一句话：

> 记忆回填流程 = 把 skill 外发生的有效上下文，补回到 truth / knowledge 体系中的流程。

## 2. 什么时候触发

优先在下面场景触发：
- 你没有使用 skill，但自己修了代码
- 你没有使用 skill，但已经实现了一个需求
- 你在别的上下文里完成了重要技术判断
- 你发现某个模块新增了稳定约束、真实行为或常见坑
- 你明确说“把这次改动记住 / 沉淀一下 / 回填到模块里”

## 3. 标准流程骨架

### 阶段 1：原始记忆采集

目标：先把这次变化中值得沉淀的信息收上来。

至少采集：
- 做了什么
- 为什么做
- 改了哪里
- 新发现了什么约束 / 行为 / 经验
- 哪些内容以后还可能复用

说明：
- 这一阶段不要求立即决定最终落点
- 重点是先把“可能值得长期保留的信息”提出来

### 阶段 2：记忆分类

目标：判断这些信息属于哪一类长期上下文。

默认分三类：
- **模块 truth**：模块职责、边界、稳定结论、长期前提
- **模块 knowledge**：模块内部机制、局部经验、耦合关系、坑点、实现依据
- **共享 knowledge**：可跨模块复用的协议、机制、约束、工程经验

说明：
- 不是所有变化都值得回填
- 默认只沉淀“下次仍值得知道”的内容，而不是完整过程日志

### 阶段 3：确定落点

目标：决定这些记忆应该写到哪里。

默认规则：
- **目标模块明确** → 直接写入对应模块
- **目标模块不明确，但内容可跨对象复用** → 写入顶层 `knowledge/`
- **目标模块不明确，且当前还无法判断最终归属** → 先进入 `intake/`，作为待归类记忆项

说明：
- 当前不新建独立的 `memory/` 顶层目录
- 待归类记忆默认复用 `intake/` 作为缓冲层

### 阶段 4：回填与压缩

目标：把原始变化压缩成未来可复用的稳定上下文。

优先写成：
- 已确认结论
- 长期约束
- 真实行为说明
- 易遗漏但代价高的信息
- 已验证有效的实施经验

避免直接写成：
- 一次性过程流水
- 临时命令
- 未验证猜测
- 只对当次排查有意义的噪音

### 阶段 5：建立引用关系

目标：让后续读取模块 / knowledge 时，能自然接到这些新记忆。

默认动作：
- 如果是模块级知识，写到模块目录内 `knowledge/`
- 如果是稳定模块结论，回写 `current-truth.md`
- 如果是跨模块知识，写到顶层 `knowledge/`
- 如果只是暂时收存，写到 `intake/` 并在 note 中标出候选挂接对象

## 4. 落点判断规则

### 4.1 写入 `modules/<module>/current-truth.md`

适用内容：
- 模块职责变化
- 稳定边界变化
- 新的输入 / 输出约束
- 新的长期前提
- 已确认的关键决策

判断口径：

> 如果这条信息以后讨论该模块时，应被当作默认成立的事实，就更适合写 truth。

### 4.2 写入 `modules/<module>/knowledge/`

适用内容：
- 模块内部机制说明
- 某个真实代码行为的解释
- 某种改法为什么不合适
- 局部耦合关系
- 调试结论、坑点、经验

判断口径：

> 如果这条信息主要服务该模块，但更像“依据 / 经验 / 机制”，就更适合写模块 knowledge。

### 4.3 写入顶层 `knowledge/`

适用内容：
- 可跨模块复用的技术机制
- 通用协议规律
- 通用工程约束
- 多个模块都适用的经验

判断口径：

> 如果这条信息不应该困在单个模块里，就更适合写共享 knowledge。

### 4.4 先进入 `intake/`

适用内容：
- 当前只知道“值得记住”，但不知道最终挂哪
- 变化来源复杂，需要后续再整理
- 当前来不及完整归类，但不想丢失上下文

判断口径：

> 如果信息值得保留，但还不适合立即定稿，就先放 intake，之后再回填。

## 5. 推荐文档承载方式

默认优先使用已有文档：
- 模块 truth → `current-truth.md`
- 模块知识 → 模块目录下 `knowledge/` 中的新知识文档
- 共享知识 → 顶层 `knowledge/` 中的新知识文档
- 待归类记忆 → `intake/<item>/note.md`

说明：
- 记忆回填流程本身不强制要求新建专用文档格式
- 它优先复用现有对象承载方式

## 6. 给模型的价值

记忆回填真正的价值不在“存起来”，而在“下次能被先读到”。

理想状态下，下次开发某个模块前，模型先读到的是：
- 当前模块 truth
- 模块 knowledge
- 最近回填的稳定经验
- 真实代码行为与历史结论之间已经校正过的上下文

这样可以减少：
- 重复分析
- 忽略隐含约束
- 因旧上下文导致的返工
