# Implement Mode

## Goal

把某项实现工作组织成可持续推进的交付单元：先接住需求来源，再按轮次锁定范围、推进实现、自检，并在需要时验收。

## Default trigger

默认由用户主动触发，例如：
- 开始做这一块
- 对某个 topic 进行代码落地
- 已经有 PRD / issue / 设计稿，直接进入实现
- 进入交付 / 实现

## Minimal structure

```text
docs-works/
└── implements/
    └── <implement-name>/
        ├── brief.md
        ├── progress.md      # 可选
        └── acceptance.md    # 可选
```

说明：
- `Implement` 默认可独立创建，不要求先存在 `Discuss`
- 如果它来自某个 discussion / topic，只在文档里记录关联来源，不强依赖目录层级
- `brief.md`：必需，用来锁定当前阶段范围，并记录需求来源
- `progress.md`：本轮跨度较大、持续多轮，或需要持续跟踪交付状态时创建
- `acceptance.md`：需要正式验收记录时再建

## Implement state model

Implement 默认使用下面这组状态来表达当前交付进度：

- `待收口上下文`：需求来源仍不足以支撑本轮范围锁定
- `待锁定`：已进入 Implement，但 `brief.md` 仍未锁稳
- `可构建`：本轮目标、范围、约束和验收标准已足够清晰，可进入 `Build`
- `构建中`：正在实现 In Scope 内容
- `待自检`：实现已基本完成，等待本轮自检
- `待验收`：需要独立验收后再收口
- `已阻塞`：当前推进受到关键阻塞项影响
- `已完成`：当前阶段 In Scope 已完成并完成检查

说明：
- `当前状态` 用于表达交付进度，不等于当前回复语气
- `Context Check / Lock / Build / Check / Accept` 是流程阶段，`当前状态` 是交付状态，二者需要同时成立
- 只有当状态达到 `可构建` 时，才建议进入 `Build`

## Section semantics

为避免 Implement 文档中的不同 section 语义重叠，核心字段按下面方式理解：

### `brief.md`
- `本轮目标`：本轮希望交付到什么程度，不等于实现清单逐条罗列
- `需求来源`：本轮实现的直接依据，可填写 discussion / topic、外部 PRD、issue、设计稿、口头需求等
- `In Scope`：本轮明确要做的内容
- `Out of Scope`：本轮明确不做的内容，用来防止范围漂移
- `实现约束`：实现过程中不能违反的边界条件
- `风险与假设`：当前已知但尚未完全消除的风险，以及被暂时接受的不阻塞假设
- `验收标准`：满足什么条件，当前阶段才算完成

### `progress.md`
- `当前状态`：当前交付状态，不是做了什么的流水账
- `当前阶段`：当前处于 `Context Check / Lock / Build / Check / Accept` 哪一步
- `本轮摘要`：对本轮推进结果的整体概述，包括偏差与变化
- `已完成`：已经落地完成、可确认的事项
- `进行中`：正在推进但尚未完成的事项
- `阻塞项`：直接阻止下一步推进的因素
- `风险 / 遗留`：当前不一定阻塞，但需要持续关注或留待后续处理的事项
- `下一步`：当前最优先的 1-3 个执行动作

### `acceptance.md`
- `验收结论`：对当前阶段是否通过验收的正式判断
- `验收范围`：本次验收覆盖到哪些内容
- `通过项`：已满足验收要求的内容
- `问题项`：本次验收中发现的缺陷或不符合项
- `遗留项`：当前接受保留到后续处理的事项
- `建议动作`：验收完成后的推荐后续动作

补充说明：
- 不要把 `本轮目标` 写成过长任务清单
- 不要让 `In Scope` / `Out of Scope` 同时覆盖同一事项
- 不要把 `阻塞项` 与 `风险 / 遗留` 混写；前者会阻止推进，后者不一定
- 不要让 `建议动作` 只是重复 `下一步` 的原话

## Context check

进入 Implement 时，默认先查当前最直接的需求来源，再开始 Lock / Build。

需求来源可以是：
- 当前 discussion / topic
- 外部需求文档 / PRD
- issue / ticket
- 设计稿 / 代码评审意见 / 现有缺陷描述

如果没有成型上下文，先判断是否仍存在会影响本轮目标、范围、实现约束或验收标准的关键待确认问题：
- 如果存在，优先回到 `Discuss` 收敛
- 如果不存在，或这些问题已被明确记录为不阻塞本轮的假设项 / 遗留项，再根据用户当前指令建立一个 `Implement`

## Round workflow

### 0. Context Check
先查本轮实现最直接相关的需求来源，并整理成可执行上下文。

如果当前仍存在会影响本轮目标、In Scope / Out of Scope、实现约束、方案选择或验收标准的关键待确认问题，默认不要直接进入 `Build`；应先补充确认、回到 `Discuss` 收敛，或先完成 `Lock` 再决定是否继续。

### 1. Lock
先锁定当前推进阶段的范围，优先创建或更新 `implements/<implement-name>/brief.md`。

至少明确：
- 本轮目标
- In Scope
- Out of Scope
- 实现约束
- 验收标准

如果这些信息仍无法稳定表达，或仍依赖关键确认，说明当前还不适合进入 `Build`；应优先补齐确认，必要时回到 `Discuss`。

### 2. Build
开始实现，只做本轮 In Scope 内的内容。

### 3. Check
实现完成后先做自检，确认：
- 是否满足本轮目标
- 是否仍在范围内
- 是否引入明显回归
- 是否达到约定的验收标准

### 4. Accept
当用户要求，或本轮风险较高时，再进入独立验收。

## Round update contract

每轮 Implement 默认按下面顺序同步更新文档：

1. 如果本轮范围边界、实现约束、关键假设或验收标准发生变化，先更新 `brief.md`
2. 根据本轮推进结果，更新 `progress.md` 的 `当前状态 / 当前阶段 / 本轮摘要 / 已完成 / 进行中 / 阻塞项 / 风险 / 遗留 / 下一步`
3. 如果进入独立验收，或需要把验收结论显式留档，再更新 `acceptance.md`
4. 回复中的当前阶段、当前状态和下一步建议，必须以后于文档中的最新结果为准

更新顺序要求：
- 先更新范围与边界，再更新进展与状态，最后更新验收记录与回复结论
- 如果本轮尚未达到 `可构建`，不要在回复或文档中越级写成可直接进入 `Build`

## Scope change rule

如果当前推进阶段的 `In Scope` / `Out of Scope` / 验收标准发生实质变化，先更新 `implements/<implement-name>/brief.md`，再继续 Build。

说明：
- 小的实现细节调整，不必回写 brief
- 只要影响本轮范围边界或完成标准，就必须先回锁范围
- 如果是否纳入本轮还不明确，先暂停扩做，必要时回到 Discuss 收敛

## Consistency rule

Implement 文档中的以下部分必须保持同步：

- `brief.md` 中的 `In Scope / Out of Scope / 实现约束 / 风险与假设 / 验收标准`
- `progress.md` 中的 `当前状态 / 当前阶段 / 已完成 / 进行中 / 阻塞项 / 风险 / 遗留 / 下一步`
- `acceptance.md` 中的 `验收结论 / 问题项 / 遗留项 / 建议动作`
- 回复中的阶段说明、状态判断与下一步建议

如果本轮新增、删除或修改了会影响是否继续 `Build`、是否进入 `Check / Accept`、或是否改变本轮边界的关键信息，必须在同一轮内同步重算：

- `brief.md` 中受影响的边界字段
- `progress.md` 中的 `当前状态 / 当前阶段 / 阻塞项 / 下一步`
- 需要时更新 `acceptance.md` 中的对应记录

禁止出现以下不一致：

- `brief.md` 中的范围已经变化，但 `progress.md` 仍按旧范围推进
- `当前状态` 仍为 `待锁定`、`待收口上下文` 或 `已阻塞`，但 `下一步` 已直接写成开始 `Build`
- 已进入 `待验收`，但回复仍把主要动作写成继续扩做而不说明原因
- 已记录关键 `阻塞项`，但 `下一步` 没有体现解除阻塞或缩小范围的动作

## Next-step suggestion rule

在 Implement 中给出 `下一步` 或回复中的下一步建议时，默认遵循下面约束：

- `待收口上下文` / `待锁定`：优先补齐上下文、完成范围锁定，不直接建议进入 `Build`
- `可构建`：优先进入 `Build`，完成 In Scope 的最小闭环
- `构建中`：优先完成当前主路径，谨慎扩展范围
- `待自检`：优先执行自检、回归检查和对照验收标准
- `待验收`：优先组织独立验收，或明确记录遗留项后再决定是否收口
- `已阻塞`：优先解除阻塞、补齐依赖、缩小范围或回到 `Discuss`
- `已完成`：优先总结结果、记录遗留，并判断是否需要开启下一轮 Implement

## Acceptance rule

独立验收默认在以下情况触发：
- 用户明确要求单独验收
- 本轮改动范围较大
- 本轮风险较高
- 需要把“实现”和“验收”分开

如果用户没有要求，且风险不高，可以只做自检。

## Reply shape

每轮 Implement 回复默认包含：
1. 本轮已落文档内容
2. 当前处于 Context Check / Lock / Build / Check / Accept 哪一阶段，以及当前交付状态
3. 当前进展、阻塞或结论
4. 下一步建议

## Reply example

示例：
1. 已把当前阶段范围锁到 `implements/<implement-name>/brief.md`，并更新 `progress.md`
2. 当前处于 Build 阶段，交付状态为 `构建中`
3. 新发现 1 个相关问题，但先记入 `风险 / 遗留`，暂不纳入当前阶段
4. 下一步建议：先完成主路径，然后进入自检

## End condition

当满足以下条件时，当前阶段 Implement 可以结束：
- 当前阶段 In Scope 内容已完成
- 已经过自检
- 需要独立验收的，已完成验收或明确遗留项
- `progress.md` 中的状态已能稳定落到 `已完成`，或明确落到 `已阻塞` / `待验收` 等待后续轮次承接

说明：
- 一个 Implement 可以包含多轮推进
- 是否结束 Implement，取决于该实现目标是否已经整体收口，而不是只看单次编码动作是否结束
- 如果本轮只是阶段性收口，应在 `progress.md` 中清楚记录当前状态和后续承接方式
