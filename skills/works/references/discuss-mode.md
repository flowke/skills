# Discuss Mode

## Goal

把需求来源、原始需求、讨论细化、上下文资料、实现思路和待确认问题整理成一个可持续演化的需求结构，为后续 Implement 和 Acceptance 提供稳定依据。

Discuss 的目标不是记录聊天过程，而是维护需求对象。

## Core rule

围绕 `topic（可选）→ 原始需求 → 讨论细化` 组织讨论：

- 默认两层：`原始需求 → 讨论细化`
- 复杂需求包可选三层：`topic → 原始需求 → 讨论细化`
- 最多三层，不继续拆 subtopic
- 原始需求不可被后续理解覆盖
- 讨论细化、上下文资料、实现思路、待确认和当前结论，都挂在对应原始需求下

## Default trigger

默认由用户主动触发；若当前已处于某个 discussion 语境内，则可在同一 discussion 中顺势延续。

进入 Discuss 的典型情况：
- 用户还在描述需求、补材料、补限制
- 用户提供产品文档、需求列表、接口文档、Figma 或项目资料
- 当前仍存在会影响实施范围、方案或验收的关键待确认
- 用户想先讨论实现思路，但还没有开始实施

## Minimal structure

### Simple discussion（无 topic）

```text
docs-works/
└── discussions/
    └── <discussion-name>/
        ├── index.md
        ├── context.md        # 可选
        └── implement/        # 进入实施时再建
```

### Topic-based discussion（有 topic）

```text
docs-works/
└── discussions/
    └── <discussion-name>/
        ├── index.md
        ├── context.md        # 可选：全局上下文
        └── topics/
            └── <topic-name>/
                ├── index.md
                ├── context.md    # 可选：topic 上下文
                └── implement/    # 进入实施时再建
```

说明：
- `index.md` 是 discussion 的唯一入口页
- `context.md` 只在有需要管理的资料时创建
- `topics/<topic-name>/index.md` 使用目录对象，不再使用 `topics/<topic-name>.md`
- `implement/` 不在 Discuss 初始阶段强制创建

## Discussion state model

Discuss 默认使用下面这组状态描述当前能推进到哪一步：

- `收集中`：仍以接收需求、材料、限制为主，尚未形成稳定需求结构
- `整理中`：已收到较多需求或资料，正在拆 topic、识别原始需求、归类上下文
- `收敛中`：已形成主要需求结构，正在推进讨论细化、实现思路和关键判断
- `待确认`：已有初步方向，但仍存在会影响下一步动作的关键未决点
- `可进入 Implement`：需求对象、范围边界、主要约束和关键上下文已足以开始实施跟踪
- `已暂停`：主要等待新材料、外部信息或用户确认，且本轮内部已无更优先的继续收敛动作
- `已完成`：当前 Discuss 目标已收口

说明：
- `可进入 Implement` 不等于已经实现，也不等于验收通过
- 如果仍存在会影响实施范围、方案、关键约束或验收标准的关键待确认，默认不进入 `可进入 Implement`

## State decision priority

当多个状态看起来同时成立时，按下面优先级判断：

1. 如果当前主要等待外部材料、外部结论或用户确认，且本轮内部已无更优先动作，标为 `已暂停`
2. 否则，如果仍存在会影响实施范围、方案或验收的关键未决点，标为 `待确认`
3. 否则，如果需求对象、范围边界、主要约束和关键上下文已足以开始实施跟踪，标为 `可进入 Implement`
4. 否则，如果已形成主要需求结构，正在推进细化或方案判断，标为 `收敛中`
5. 否则，如果正在拆 topic、识别原始需求或整理资料，标为 `整理中`
6. 否则，标为 `收集中`

## Topic rule

### When to create topic

只有当需求包覆盖多个页面、模块、业务域或版本内功能区时，才创建 topic。

可以创建 topic 的信号：
- 产品文档明确分为多个页面 / 模块
- 需求天然属于不同业务域
- 不同部分有独立上下文、独立待确认、独立实施节奏
- 一个 discussion 的需求地图已经过长，按页面 / 模块拆分后更易推进

不要因为单个原始需求变长就自动创建 topic；产品文档里的长描述仍属于原始需求 / 原始描述，后续讨论循环新增的补充才属于讨论细化。

### Topic meaning

- topic = 页面 / 模块 / 业务域 / 版本内功能区
- topic 下面可以有多个原始需求
- topic 不默认等于原始需求
- 最多三层：`topic → 原始需求 → 讨论细化`

### Topic relation to index

`discussion/index.md` 必须保留 topic 地图：
- topic 名称
- topic 记录位置
- 当前状态
- 当前结论 / 卡点
- 下一步

新建、合并、删除 topic 或 topic 状态变化时，必须同轮更新 `discussion/index.md`。

## Requirement block rule

无论在 `discussion/index.md` 还是 `topics/<topic-name>/index.md` 中，原始需求都用稳定编号维护，例如 `R1`、`R2`、`R3`。

每个原始需求块默认使用下面字段：

```md
### R1. <原始需求标题>
- 原始需求 / 原始描述：
  - 
- 讨论细化：
  - 暂无。
- 相关上下文：
  - 
- 实现思路 / 方案预对齐：
  - 暂无。
- 待确认：
  - 
- 当前结论：
  - 暂按原始需求记录，尚未完成讨论细化。
```

字段语义：
- `原始需求 / 原始描述`：保留需求来源中的原始诉求与原始细节；产品文档里的字段、文案、状态和流程，即使很细，也仍放这里
- `讨论细化`：只记录后续讨论循环中新增确认、澄清或补充出来的边界、规则、例子、默认行为、优先级和依赖；初始导入时通常为 `暂无`
- `相关上下文`：列出直接影响该需求理解或实现的 context 条目，必要时链接到 context 文件

## Source ownership rule

区分 `原始需求 / 原始描述` 和 `讨论细化` 时，按来源判断，而不是按内容是否详细判断：

- 产品文档、用户初始输入、issue、设计稿说明中已经存在的内容，一律归入 `原始需求 / 原始描述`
- 后续讨论循环中新确认、新澄清、新补充的内容，才归入 `讨论细化`
- AI 根据经验推断出的内容，不能直接写入 `讨论细化` 或 `当前结论`；如确有价值，应写入 `待确认`，或标成 `AI 初步建议（待确认）`
- 初始导入产品文档时，默认只拆 topic、提取原始需求、记录 context、识别明显待确认；不要提前生成讨论细化或实现方案

- `实现思路 / 方案预对齐`：记录 Discuss 阶段聊出的实现方案、技术路径、取舍或候选方案；不等于已经开始实施
- `待确认`：只记录仍会影响该需求继续推进、实施范围、方案或验收的关键未决点
- `当前结论`：记录该需求当前已经可以成立的理解和默认推进口径

## Context management rule

当用户提供接口文档、Figma、项目资料、现有代码说明、截图、链接或其他辅助资料时，默认作为 context 管理。

context 不应只是链接堆积。每个 context 条目尽量记录：

```md
### C1. <资料名>
- 类型：产品文档 / 接口文档 / Figma / 项目资料 / 代码说明 / 其他
- 来源：
- 关联范围：全局 / Topic / R1 / R2
- 当前状态：未消化 / 已消化 / 已吸收进需求 / 已吸收进实现 / 暂不采用
- 摘要：
- 对需求 / 实现的影响：
```

放置规则：
- 影响整个 discussion 的资料，放 `discussion/context.md`
- 只影响某个 topic 的资料，放 `topics/<topic-name>/context.md`
- 只影响实施方案或技术落地的资料，可在进入 Implement 后转写或链接到 `implement/context.md`
- 原始需求块中的 `相关上下文` 只放摘要和引用，不要复制大段资料

## Implementation idea rule

Discuss 阶段允许讨论实现思路，但必须和正式实施区分。

规则：
- 实现思路放在原始需求块的 `实现思路 / 方案预对齐`
- 标明候选方案、当前倾向、待确认和已确认部分
- 不要因为讨论了实现思路就写成已经进入 Build
- 进入 Implement 时，已确认的实现思路应承接到 `implement/progress.md` 的 `当前实施口径` 或 `implement/context.md`

## Default discussion loop

默认按下面顺序循环推进：

1. **接收输入**：判断本轮是需求来源原文、新增原始需求、讨论循环产生的讨论细化、上下文资料、实现思路，还是待确认反馈
2. **归位结构**：放到对应 discussion / topic / 原始需求块 / context 文件
3. **收敛判断**：更新待确认、当前结论和状态
4. **重算下一步**：判断继续 Discuss、补资料、确认关键问题，还是可进入 Implement
5. **同轮给方案**：如果已经缩到 1-3 个可拍板问题，直接给推荐方案和必要备选

## Focus selection rule

当 discussion 下存在多个 topic 或多个原始需求时，默认先判断：是否已经可进入 Implement。

- 如果可以进入 Implement：建议切到 Implement，并说明实施对象是 discussion 还是 topic
- 如果不能进入 Implement：继续推进最阻塞下一步的 topic / 原始需求 / 全局待确认
- 如果关键问题跨多个 topic：停在 `discussion/index.md` 层处理，不强塞进某个 topic

回复中只需简要说明本轮焦点，例如：
- `这轮先推进 Topic A，因为它还有影响实施范围的关键待确认。`
- `这轮先停在 index 层处理跨 topic 的接口约束。`

## Round update contract

每轮 Discuss 默认按下面顺序更新：

1. 如果新增页面 / 模块 / 业务域，更新 topic 地图，必要时创建 topic 目录
2. 如果新增原始需求，创建或更新对应 `R#` 需求块
3. 如果是需求来源原文里的详细描述，写入 `原始需求 / 原始描述`；如果是后续讨论循环新增确认的补充，才写入 `讨论细化`
4. 如果新增资料，写入对应 context，并在相关需求块中引用
5. 如果新增实现讨论，写入对应需求块的 `实现思路 / 方案预对齐`
6. 更新 `待确认`，删除已确认项，避免已降级问题继续阻塞
7. 更新 `当前结论`
8. 更新 discussion / topic 的 `当前状态`
9. 更新 `下一步`

禁止只更新局部细节而不重算状态和下一步。

## Consistency rule

必须保持同步：
- `discussion/index.md` 的 topic 地图 / 全局待确认 / 当前状态 / 下一步
- `topics/<topic-name>/index.md` 的需求地图 / topic 待确认 / 当前状态 / 下一步
- 相关 context 的消化状态和影响说明
- 原始需求块中的 `相关上下文` 引用

禁止出现以下不一致：
- topic 已创建，但 discussion index 没有入口
- 原始需求 / 原始描述被后续讨论细化改写，导致看不出原始诉求
- context 已影响实现，但没有在需求块或 implement context 中引用
- Discuss 仍有关键待确认，却建议直接 Build
- 实现思路尚未确认，却写成当前结论或实施完成
- 已降级为后续版本项的问题仍保留在关键待确认中

## Next-step suggestion rule

在 Discuss 中给出下一步时：
- 如果存在关键待确认，优先建议完成确认、补资料或缩小范围
- 如果缺少上下文资料，优先建议补资料或明确默认假设
- 如果需求结构清晰且阻塞项已清除，建议进入 Implement，并指出实施对象
- 不直接建议开始编码，除非已经进入 Implement 且 progress 具备可执行子项
- 如果能形成推荐方案，默认同轮给出推荐方案，不预告下一轮再给

## End condition

当满足以下任一条件时，可以结束当前 Discuss：
- 需求结构、关键上下文、待确认和当前结论已足以进入 Implement
- 当前主要等待新材料、外部信息或用户确认，可先置为 `已暂停`
- 当前 Discuss 目标已完成，可置为 `已完成`
