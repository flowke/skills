# Discuss Mode

## Goal

把讨论中的需求、材料和问题整理成一个持续演化的讨论文档，并在需要时重组为主题化结构。

## Core rule

先统一承接，必要时再按主题重组。

## Invocation preference

默认优先显式触发 Discuss，不主动自动进入。
只有当用户已经明显在当前 discussion 语境内持续推进时，才顺势延续。

## Facilitation style

Discuss 不只整理内容，还要承担轻度主持职责：
- 主动聚焦当前最值得讨论的问题
- 发现混线时主动分流
- 发现缺口时主动补洞
- 每轮给出下一步建议

回复要求：
- 保持简洁
- 没有特别必要时，不解释长原因
- 一次优先给 2-3 个可选项或建议
- 避免给过长清单

## Default start

当用户刚开始描述需求、补充说明或提供材料时：
- 先维护一个统一讨论页
- 不要急着拆成多个主题文档
- 不要按聊天时间顺序堆积内容

## How to organize the discussion page

优先按讨论对象组织内容；如果对象边界还不清楚，再先按较粗粒度整理。

统一讨论页默认使用 `references/unified-discussion-template.md`。
topic 页默认使用 `references/topic-template.md`。
拆分迁移时优先参考 `references/migration-rules.md`。
文档组织结构优先参考 `references/document-organization.md`。

统一讨论页至少应包含：
- 讨论目标
- 当前输入
- 当前理解
- 讨论板块
- 跨板块问题
- 待确认问题
- 下一步建议

## When topic boundaries are not explicit

如果用户没有明确区分主题：
- 不要强行拆主题
- 先稳定记录统一讨论页
- 保持内容可重组

## Triggers for modular reorganization

从统一页切到分主题页，我的默认判断是：**当统一页已经不足以低成本承接讨论时，就切。**

出现以下任一情况时，开始按主题或对象重组内容：
- 用户明确要求分主题管理
- 讨论中已经出现多个相对独立的对象
- 某部分内容有自己独立的目标、问题、材料或约束
- 某部分内容后续会独立推进
- 同一页里不同对象的内容开始频繁混线，阅读和续写成本明显上升

默认不要为了形式整齐过早拆分；只有在拆分后能明显提升讨论清晰度、续写效率或后续推进效率时才切。

## Switching threshold

统一页适合：
- 需求还在发散期
- 主题边界还不稳定
- 大部分问题仍然是全局性的

分主题页适合：
- 已经能稳定说出 2 个以上讨论对象
- 每个对象下面都有各自的结论、问题或材料
- 后续大概率会分开继续讨论或独立推进

一个简单判断句：
**如果继续写在同一页里会越来越混，就该切；如果拆了之后还会频繁合并着讨论，就先别切。**

## How to reorganize by topic

分主题时，默认采用：**一个总页 + 多个主题子块 / 子页**。

处理方式：
- 保留统一讨论总页，作为总入口
- 在总页中维护各主题的摘要子块
- 当某个主题内容继续长大时，再把该主题展开为独立子页
- 基于已有内容重组，不要从头重写

每个主题内部整理：
- 当前目标
- 当前结论
- 待确认问题
- 相关材料
- 下一步

## Promote to independent discussion root

如果某个主题非常复杂，可以把它从当前讨论树中拆出来，成为一个新的 `<discussion-root>`。

这不是普通的主题子页展开，而是一次新的讨论树分叉。

唯一默认触发条件：
- 用户主动明确要求“拆成新的”

没有这个主动指令时，不要自作主张把某个主题升级为新的 `<discussion-root>`。

拆出去后：
- 原 root 删除与该主题相关的所有内容
- 详细讨论整体迁移到新的 `<discussion-root>` 中
- 默认不在原 root 保留摘要、入口或关系说明
- 是否保留跨主题关联，由用户另行明确要求

## Cross-topic section

涉及多个主题的内容不要硬塞进某个主题，单独维护“跨主题问题”区域，用来记录：
- 依赖关系
- 公共约束
- 统一规则
- 全局待确认问题

## Page relationship

默认结构：
- 一个统一讨论总页：承接全局背景、总体状态、主题导航、跨主题问题
- 多个主题子块：先在总页里做摘要
- 多个主题子页：只在某主题内容明显变大后再展开

总页负责：
- 当前讨论主题
- 主题列表与简短摘要
- 跨主题问题
- 当前优先级
- 下一步建议

子页负责：
- 某个主题的详细讨论
- 该主题自己的材料、问题、结论与下一步

默认不要一开始就创建很多子页；先有总页，按需要生长出子页。


## Documentation rhythm

- 边讨论边落文档
- 每轮回复开头先简短说明本轮哪些内容已经落入文档
- 只说明已落文档的内容，不展开过程

## Index update priority

每轮讨论默认优先更新当前 discussion 的 `index.md`，优先顺序如下：
1. 当前输入
2. 当前理解
3. 讨论板块
4. 待确认问题
5. 下一步建议

条件更新：
- `主题导航`：只有已经出现多个主题时才更新
- `来源说明`：只有这个 discussion 是拆出来的新 discussion 时才写
- `跨板块问题`：只有真的出现跨主题约束、依赖或全局问题时才更新

## End condition

当满足以下任一条件时，可以结束本轮 Discuss：
- 已经形成较清晰的讨论结构
- 已经可以进入后续交付规划
- 当前主要缺少新材料或用户确认
