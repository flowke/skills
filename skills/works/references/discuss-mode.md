# Discuss Mode

## Goal

把讨论中的需求、材料和问题整理成一个持续演化的讨论结构，并在必要时拆出 topic 子页。

## Core rule

先维护总页，并始终让总页能表达当前主线；只有当某个板块已足够独立时，才拆出 topic。

## Default trigger

默认由用户主动触发；若当前已处于某个 discussion 语境内，则可在同一 discussion 中顺势延续。

## Minimal structure

默认只使用下面这套最小结构：

```text
docs-works/
└── discussions/
    └── <discussion-name>/
        ├── index.md
        └── topics/
            └── <topic-name>.md
```

说明：
- `index.md`：总页，负责承接全局背景、当前理解、关键未决点和全局下一步
- `topics/*.md`：子页，只在某个板块已经足够独立时才创建

## Naming

- discussion 目录：`docs-works/discussions/<discussion-name>/`
- topic 子页：`docs-works/discussions/<discussion-name>/topics/<topic-name>.md`
- 名称使用短、稳定、可读的 kebab-case

## Discussion state model

Discuss 默认使用下面这组状态来描述当前进度：

- `收集中`：当前仍以补充输入、材料、限制为主，尚未形成稳定的问题框架
- `收敛中`：已识别主要问题，正在推进关键判断，但还未形成稳定的下一步边界
- `待确认`：已有初步方向，但仍存在会影响下一步动作的关键未决点
- `可进入 Implement Lock`：本轮目标、范围边界、主要约束已足够清晰，可转入 `Implement` 先锁定 brief
- `已暂停`：当前主要等待新材料、外部信息或用户确认，且本轮内部已无更优先的继续收敛动作
- `已完成`：当前 Discuss 目标已收口

说明：
- `Discuss` 中的状态用于表达“当前能推进到哪一步”，而不是表达聊天语气
- `可进入 Implement Lock` 只表示可以进入 `Implement` 做范围锁定，不等于已经可以直接开始 `Build`
- 只要仍存在会影响下一步动作的关键未决点，默认不进入 `可进入 Implement Lock`

## State decision priority

当多个状态看起来同时成立时，按下面优先级判断：

1. 如果当前主要等待外部材料、外部结论或用户确认，且本轮内部已无更优先的继续收敛动作，标为 `已暂停`
2. 否则，如果仍存在会影响下一步动作的关键未决点，标为 `待确认`
3. 否则，如果本轮目标、范围边界和主要约束已足够清晰，且剩余问题已被降级为非阻塞项，标为 `可进入 Implement Lock`
4. 否则，如果已识别主要问题并正在推进关键判断，标为 `收敛中`
5. 否则，标为 `收集中`

## Board and topic relation

为避免 `当前板块` 与 `topic` 混用，默认按下面方式理解：

- `板块`：`index.md` 中的逻辑分区，用来承接某一条讨论主线
- `topic`：当某个板块已具备独立目标、独立问题和独立后续推进时，从该板块提升出的独立文档
- 未提升为 `topic` 前，内容只保留在 `index.md` 的对应板块中
- `index.md` 默认至少保留 1 个板块；如果当前还没有自然分区，就使用一个总板块承接当前主线

## Default discussion loop

采用**中强主持**，但保持自然对话感。

默认按下面顺序循环推进：
1. **先定焦**：先判断当前 discussion 是否已经可进入 `Implement Lock`；如果还不能，再看这轮最自然该继续推进哪个问题、板块或 topic。
2. **再推进 1-3 个关键问题**：默认优先推进 1 个问题；当多个问题高度相关、一起推进更高效时，可在同一轮推进 2-3 个问题。
3. **做阶段总结**：在自然节点收一下当前理解、关键未决点和已形成的局部结论。
4. **更新讨论文档**：把稳定内容写入当前 discussion / topic，把未收口内容按阻塞性分类。
5. **给出下一步聚焦**：指出下一轮最值得继续推进的问题，并附 1-3 条具体建议；如果已经能形成推荐方案，则在本轮直接给出“推荐方案 + 备选 + 需要确认的一句话”。

## Focus selection rule

当一个 discussion 下已经出现多个 topic / 板块时，默认先只判断一件事：**当前是否已经可进入 `Implement Lock`**。

- 如果**已经可进入 `Implement Lock`**：不用继续扩展 Discuss，直接建议进入 `Implement`
- 如果**还不能进入 `Implement Lock`**：就继续推进 Discuss

也就是说，默认不是先做复杂的 topic 优先级计算，而是先看：
- 当前是否还有关键待确认问题
- 这些问题是否仍会影响下一步动作
- 如果会影响，就继续推进，直到能进入 `Implement Lock`

当还不能进入 `Implement Lock`，而 discussion 下又有多个 topic 时，默认用下面这个简单规则：

1. **继续当前正在推进的 topic / 板块**
   - 如果上一轮已经有明显主线，就先沿着那条主线继续，不必频繁切换
2. **如果当前 topic 已经收口或不再阻塞，就回到 `index.md` 看剩余关键待确认问题**
   - 再从这些问题里选下一个最自然需要继续推进的 topic
3. **如果某个关键待确认问题明显属于某个 topic，就继续那个 topic**
   - 不需要额外建立复杂排序，只要能解释“为什么还不能进入 `Implement Lock`”即可
4. **如果关键问题跨多个 topic，就先停在 `index.md` 层继续推进**
   - 不急着强行落到单个 topic

默认目标很简单：
- 不是为了证明“哪个 topic 更高级”
- 而是为了把 discussion 继续推进到 `可进入 Implement Lock`

## Focus selection output rule

当本轮在多个 topic 之间决定继续推进哪个时，回复里只需要补一句简短说明，例如：

- `这轮先继续 topic A，因为它还卡着进入 Implement Lock 前的关键确认。`
- `这轮先不进入单个 topic，先在 index 层把剩余关键待确认问题收一轮。`

说明到这个粒度就够了，默认不需要展开复杂优先级分析。

## Section semantics

为避免不同 section 之间语义重叠，Discuss 文档中的核心字段按下面方式理解：

- `当前输入`：记录本轮新增的事实、材料、限制、反馈和问题，不承担分析和决策
- `当前理解`：记录基于当前输入形成的整体判断，包括问题本质、边界、分歧和当前优先级
- `当前板块`：记录总页当前有哪些讨论主线；每个板块至少包含 `当前结论 / 待确认 / 下一步`
- `当前结论`：记录某个板块或 topic 当前已经可以成立的局部结论，用于承接后续推进
- `板块内待确认`：只记录该板块的局部未决点，不自动等同于全局阻塞项
- `关键待确认问题`：只记录仍会影响整个 discussion 下一步动作的关键未决点；这是判断能否进入 `Implement Lock` 的主依据
- `开放问题（不阻塞本轮）`：记录尚未最终拍板，但已被接受为当前阶段不阻塞的假设、默认值、后续版本项或观察项
- `板块内下一步`：记录如果继续推进该板块，最合适的局部下一动作是什么
- `下一步建议`：记录从整个 discussion 视角出发，当前最值得优先推进的 1-3 个动作，属于全局下一步

补充说明：
- 不要把 `当前输入` 写成分析摘要
- 不要把 `当前理解` 直接写成具体动作清单
- 不要让 `当前结论` 只是重复 `当前理解` 的原话
- 纯局部、且不影响全局推进的未决点，不必抬升到全局 `关键待确认问题`
- 当板块内下一步与全局下一步冲突时，以全局 `下一步建议` 为当前轮的优先级输出
- 当某项问题已被接受为当前阶段不阻塞的默认实现约定、假设项或后续版本项时，不应继续留在 `关键待确认问题`

## Round update contract

每轮 Discuss 在形成当前结论后，默认按下面顺序同步更新文档：

1. `当前输入`：补入本轮新增材料、限制、问题或反馈
2. `当前理解`：更新对问题、目标、边界和分歧点的当前判断
3. `当前板块`：同步更新对应板块下的 `当前结论 / 待确认 / 下一步`
4. `关键待确认问题`：收敛并重排仍会影响全局下一步动作的关键未决点，删除已确认项
5. `开放问题（不阻塞本轮）`：补入已被接受为不阻塞的默认值、假设项、后续版本项和观察项
6. `当前状态`：根据最新判断，更新当前所处状态
7. `下一步建议`：基于最新 `关键待确认问题`、`开放问题（不阻塞本轮）` 和 `当前状态` 重新给出下一步动作

更新顺序要求：
- 先更新事实和判断，再更新未决点分类，再更新状态，最后更新动作建议
- `下一步建议` 必须以后于 `关键待确认问题` 和 `当前状态` 的最新结果为准

## How to work

默认按下面方式落结构：
1. 先用 `index.md` 承接当前 discussion
2. 新输入先落到总页，不急着拆 topic
3. 总页默认至少保留 1 个 `当前板块`；如果还没有自然分区，就先用一个总板块承接
4. 当某一板块已经能独立承接“目标 + 问题 + 后续推进”时，再创建 topic 页
5. topic 创建后，总页保留总览，topic 承接细节
6. 如果某个新确认的内容、结论、限制、默认约定或优先级变化会同时影响多个板块 / topic，默认只在同一个父 topic / 父板块内部联动更新；不要只更新当前正在编辑的那一页，但也不要自动跨到其他父 topic，除非用户主动要求，再把对应范围与 index.md 一起同步到最新语义
7. 只有用户明确要求“拆成新的”时，才把 topic 升级成新的 discussion

## Consistency rule

Discuss 文档中的以下部分必须保持同步：

- `当前理解`
- `当前板块`
- `关键待确认问题`
- `开放问题（不阻塞本轮）`
- `当前状态`
- `下一步建议`

若 discussion 已拆出多个 topic，则一致性要求默认同时覆盖：

- `index.md` 中对应的全局字段
- 所有被本轮变化影响到的 `topics/*.md`

如果某项内容的确定会同时改变多个 topic 的目标、边界、结论、待确认问题、状态或下一步建议，默认只同步更新同一个父 topic / 父板块下受影响的这些 topic，而不是只更新当前正在编辑的单个 topic；若会跨到其他父 topic，除非用户主动要求，否则不要自动扩散更新。

如果本轮新增、删除或修改了会影响下一步动作的关键信息，必须在同一轮内同步重算：

- `关键待确认问题`
- `开放问题（不阻塞本轮）`
- `当前状态`
- `下一步建议`

如果这些变化跨越同一个父 topic / 父板块下的多个 topic，则以上重算范围也必须同步扩展到这些受影响的 topic 与 `index.md`，禁止只在单个 topic 内局部收口；若变化会牵动其他父 topic，则默认先保持边界，除非用户主动要求跨父级同步。

禁止出现以下不一致：

- `关键待确认问题` 中仍存在关键阻塞项，但 `下一步建议` 已写成可进入 `Implement Lock` 或开始实施
- `当前状态` 仍为 `待确认` 或 `已暂停`，但 `下一步建议` 已越级到实施动作
- `当前状态` 已为 `可进入 Implement Lock`，但 `当前理解` 中仍写着“本轮目标 / 范围边界 / 主要约束尚未明确”
- `当前理解` 已发生变化，但 `当前板块` 或 `下一步建议` 仍沿用上一轮判断
- 某项问题已被写成默认值、假设项或后续版本项，却仍同时保留在 `关键待确认问题`
- 在同一个父 topic / 父板块内部，某项内容已在 topic A 中被确认、降级或改写，但同范围内 topic B / `index.md` 中对应语义仍停留在旧版本
- 未经用户主动要求，就把某个确认结果自动扩散到其他父 topic，导致原本应保持边界的讨论被提前改写

## Next-step suggestion rule

在 Discuss 中给出 `下一步建议` 时，先判断当前是否仍存在关键待确认问题：

- 如果存在，下一步建议应优先指向：继续收敛讨论、补材料、补限制、完成关键确认、拆清范围边界
- 如果不存在，且当前目标、范围边界和主要约束已经足够清晰，才可以建议进入 `Implement Lock`
- 当建议进入 `Implement Lock` 时，应同时明确：当前剩余未决项中，哪些已被降级为非阻塞假设、默认实现约定或后续版本项
- `Discuss` 中的下一步建议默认不直接给出 `开始 Build`、`开始编码`、`开始实施` 这类越级动作
- 如果当前已收敛到 1-3 个可直接拍板的关键问题，默认应在同一条回复里直接给出推荐方案，而不是只描述“下一轮要聊什么”

说明：
- `Discuss` 最多把工作推进到“可进入 `Implement` 做范围锁定”
- 是否进入 `Build`，应在 `Implement` 中经过 `Lock` 后再决定
- 只有同时满足以下条件，才可写“可进入 Implement Lock”或建议进入 `Implement Lock`：
  - 本轮目标已明确
  - 范围边界已可初步表达
  - 主要约束已明确
  - 不存在会阻止锁定 brief 的关键待确认问题
  - 若仍有未决项，这些项已被明确归类为非阻塞假设、默认实现约定或后续版本项

## Same-turn proposal rule

当 Discuss 已经缩到“再确认 1 步就能继续”的状态时，默认采用**同轮给方案**，而不是**预告下一轮再给方案**。

默认做法：
- 直接给出 1 个`推荐方案`，说明为什么它是当前首选
- 必要时再给 1-2 个`备选方案`，只写关键差异、代价或适用前提
- 最后把用户需要确认的话压缩成一句可直接回复的话，例如“如果你接受默认按 A，我就把 B 记为后续版本项”

默认禁止：
- 只写“下一步继续讨论 X”而不提供任何可拍板内容
- 只写“如果你愿意，我下一条再给建议方案”这类额外来回
- 明明已有足够上下文，却把可直接判断的问题继续悬空

只有在以下情况，才允许不在本轮直接给推荐方案：
- 当前材料仍不足以支撑合理推荐
- 推荐方案一旦给错，会明显改变范围、约束或验收标准
- 用户明确要求先不要给方案，只做整理或记录

## Reply shape（强约束版）

当 Discuss 已经进入“可直接继续推进，只差用户拍板或小范围确认”的状态时，回复默认压缩成下面 4 段，不额外铺长过程：

1. `已落文档`：一句话说明本轮更新到了哪个 discussion / topic 文档
2. `当前阶段 / 状态`：例如 `Discuss / 待确认`、`Discuss / 收敛中`
3. `当前唯一关键结论或阻塞`：只写当前最影响下一步动作的那 1 条，不把所有历史问题重新摊开
4. `下一步建议`：默认写成 `推荐方案 + 备选 1-2 条 + 一句可直接确认的话`

默认要求：
- 如果这一轮真正只剩 1 个关键拍板点，就不要再并列抛出多个同权问题
- 如果备选方案价值很低，可以只写推荐方案，不强制凑满备选
- 如果当前没有足够上下文支撑推荐方案，才退回普通的“继续收敛 / 补材料”回复
- 除非用户明确要求展开过程，否则不要在这 4 段之外继续铺分析长文

## Reply examples

### 正例（推荐）

适合在当前已收敛到一个关键拍板点时使用：

```md
已把这轮讨论收敛结果更新到 `docs-works/discussions/<discussion-name>/index.md`。

当前处于 Discuss / 待确认。
现在真正剩下的关键确认只有 1 个：默认模板块是否承接固定结构块集合。

下一步建议：
- 推荐方案：默认模板块先承接固定结构块集合。这样首版的数据模型、编辑约束和导出路径最稳定，能先把主链路收住。
- 备选 1：模板块只提供空骨架，不承接固定结构集合。灵活性更高，但首版更容易把结构约束做散。
- 备选 2：拆成“固定模板块 / 自由模板块”两层。表达更清楚，但会提高首版模型复杂度。

你只需要确认一句：如果首版优先稳定落地，我建议默认按推荐方案推进。
```

### 反例（避免）

下面这种写法默认不推荐：

```md
已更新讨论文档。

当前还剩 1 个关键问题：默认模板块是否作为固定结构块集合承接。

下一步建议：
- 下一轮继续讨论这个问题。
- 如果你愿意，我下一条直接给你建议方案。
```

原因：
- 它只复述了“接下来聊什么”，没有提供可直接拍板的内容
- 它额外制造了一轮往返，降低沟通效率
- 在已有足够上下文时，没有完成 Discuss 应做的“同轮给方案”

## Topic creation threshold

默认先只更新 `index.md`，不要急着创建 `topics/*.md`。

只有当某一板块已经能独立承接下面三件事时，才创建 topic 页：
- 独立目标
- 独立问题 / 结论 / 材料
- 后续独立推进

如果一个部分还不能独立承接“目标 + 问题 + 后续推进”，就先留在 `index.md`。

## Promote to a new discussion

只有当用户主动明确要求“拆成新的”时，才把某个 topic 拆成新的 discussion。

拆出去后：
- 原 discussion 删除与该 topic 相关的内容
- 默认不保留摘要、入口或关系说明
- 新 discussion 继续使用同样的最小结构

## Minimal templates

### `index.md`
