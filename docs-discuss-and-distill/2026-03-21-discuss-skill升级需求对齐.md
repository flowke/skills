# discuss skill 升级需求对齐

## 当前阶段
- 需求对齐

## 当前真相
- 讨论主题：升级 `discuss-and-distill` skill 的职责范围与主流程。
- 当前目标：把它从偏“模糊话题讨论收敛”的 skill，升级为可承接复杂需求讨论、对齐、规划衔接与沉淀的 skill。
- 核心定位：`discuss-and-distill` 升级后，不是普通讨论 skill，而是一个**复杂需求讨论与沉淀编排器**。
- 职责范围（当前已确认）：
  - 接收复杂需求
  - 引导对齐
  - 补遗漏 / 查冲突
  - 在需求稳定后衔接到代码落地规划
  - 在后续回归时维护同一份 current truth
- 轻重分流的判断本身是成立的：
  - 轻量需求：更适合直接走 agent plan / 执行流
  - 复杂需求：更适合进入 discuss 这类讨论与沉淀流程
- 但已确认：**skill 本身不负责做轻重分流**。
  - 理由：一旦进入这个 skill，就默认当前已经是值得讨论、对齐和沉淀的场景；分流属于 skill 外部的使用判断，而不是 skill 内部第一步。
- 已确认需要有“**一致性复盘 / 规划前检查**”这一环：
  - 至少在进入代码落地规划之前必须发生
  - 作用是把“看起来讲清楚了”变成“足够稳，可以规划了”
- 这轮一致性复盘至少要覆盖：
  - 冲突检查
  - 相互影响检查
  - 遗漏项检查
  - 待决策点检查

## 待确认项
- “一致性复盘”在流程里更适合：
  - 作为一个固定显式阶段
  - 还是作为进入规划前必须完成、但不一定总要显式命名的检查动作
- “回归”在这个 skill 里的定义边界还未最终拍板：
  - 更偏需求—规划—实现的一致性回看
  - 是否还包含实现后验收、用户补充信息后的重新对齐等场景
- “背景资料提交”是否要被定义为一个明确阶段，还未最终确认。
- 这个 skill 的完整阶段链路是否固定为：
  - 背景资料提交
  - 需求对齐
  - 一致性复盘
  - 代码落地规划
  - 回归
  - 持续沉淀
  目前倾向是这条链路成立，但仍需继续对齐细节。

## 未决问题
- 当前最关键：
  - “回归”在升级后的 discuss skill 中，到底精确定义成什么。
- 其他：
  - 一致性复盘的呈现方式，是显式阶段、隐式必做，还是默认显式但可轻量化。
  - 背景资料提交是否要单独作为主流程阶段写进 skill。
  - 进入代码落地规划后，skill 的职责是到规划为止，还是继续覆盖到规划后的回归 / 回看。

## 冲突与影响检查
- 当前没有发现已确认项之间的硬冲突。
- 已识别到一个容易混淆的点：
  - “轻重分流成立” 与 “skill 要不要负责分流” 是两回事。
  - 当前已明确：前者成立，后者不成立。
- 已识别到一个关键设计点：
  - “一致性复盘”不能缺席，但它在交互上要不要被设计成强显式阶段，仍需权衡自然度与稳定性。

## 下一步聚焦
1. 先把“回归”的定义拍板。
   - 建议：优先明确它是否只指“需求—规划—实现的一致性回看”，还是还包含验收 / 重新对齐等更广的场景。
   - 建议：如果有多种语义，先收成 1 个主定义 + 1 组可包含的扩展场景，避免 skill 过宽。
2. 再确定“一致性复盘”的交互形态。
   - 建议：围绕“固定显式阶段 / 隐式必做检查 / 默认显式但允许轻量化”三种方式做取舍。
   - 建议：重点看哪种方式最不容易漏掉进入规划前的必要检查。
3. 最后再收完整主流程。
   - 建议：等“回归”和“一致性复盘”两个点拍板后，再决定是否把“背景资料提交”写成明确阶段。
   - 建议：届时再统一整理成 skill 的正式升级需求，而不是边讨论边直接改文件。
