# discuss skill 升级需求对齐

## 当前阶段
- 需求对齐

## 当前真相
- 讨论主题：升级 `discuss-and-distill` skill 的职责范围与主流程。
- 当前目标：把它从偏“模糊话题讨论收敛”的 skill，升级为可承接复杂需求讨论、对齐、规划衔接与沉淀的 skill。
- 核心定位：`discuss-and-distill` 升级后，不是普通讨论 skill，而是一个**复杂需求讨论与沉淀编排器**。
- 职责范围（当前已确认）：
  - 接收复杂需求
  - 引导对齐
  - 补遗漏 / 查冲突
  - 在需求稳定后衔接到代码落地规划
  - 覆盖到代码完成后的回归阶段
  - 在代码完成后的回归阶段维护同一份 current truth
  - 根据回归结果更新问题、偏差与后续待决项
- 轻重分流的判断本身是成立的：
  - 轻量需求：更适合直接走 agent plan / 执行流
  - 复杂需求：更适合进入 discuss 这类讨论与沉淀流程
- 但已确认：**skill 本身不负责做轻重分流**。
  - 理由：一旦进入这个 skill，就默认当前已经是值得讨论、对齐和沉淀的场景；分流属于 skill 外部的使用判断，而不是 skill 内部第一步。
- 已确认需要有“**一致性复盘 / 规划前检查**”这一环：
  - 至少在进入代码落地规划之前必须发生
  - 作用是把“看起来讲清楚了”变成“足够稳，可以规划了”
  - **默认作为显式环节保留，但允许按场景轻量化**
  - 无论轻量还是完整，**都不能跳过**
- 这轮一致性复盘至少要覆盖：
  - 冲突检查
  - 相互影响检查
  - 遗漏项检查
  - 待决策点检查
- 已确认“**回归**”的定义：
  - 指**代码写完之后进行回归**
  - 不指广义的“重新进入需求对齐”或“任意阶段回看”
  - 理由：这里要保留工程语义，避免把“回归”扩成泛化的返场机制
- 已确认这个 skill 的职责边界：
  - **覆盖到代码完成后的回归阶段**，否则会与前面的对齐 / 规划阶段形成割裂
  - 但重点是**回归结果的沉淀、问题回写与 truth 闭环**
  - **不负责回归执行本身**，也不替代测试 / 验收动作
- 已确认“**背景资料提交**”的定位：
  - **不单列为固定主流程阶段**
  - 更适合作为进入需求对齐前或对齐过程中**按需触发的输入 / 补充动作**
  - 理由：它更像资料准备与上下文补齐，不是一个必须独立推进的收敛阶段
- 已确认完整主流程的设计方式：
  - 采用**标准骨架 + 按场景裁剪**，而不是每次都要求完整经过固定全链路
  - 推荐主骨架为：**需求对齐 → 一致性复盘 → 代码落地规划 → 回归结果沉淀 → 持续沉淀**
  - 其中“背景资料补充”是按需动作，不单列为固定主阶段
  - 其中“回归结果沉淀”发生在**代码完成后的回归阶段**，负责把结果、问题、偏差回写到同一份 current truth
- 已确认产出文件组织的基础模型：
  - 从“**一主题一文件**”升级为“**一个主题一个目录**”
  - 每个主题目录中维护一份**canonical current-truth 主文档**
  - 再按需补充 `planning`、`regression` 等阶段文件
  - 理由：既保留同一主题持续维护的连续性，又避免把 current truth、规划细节和回归沉淀全部混在一份超长文档里
- 已确认主题目录中的最小文件集合：

  - **按需**：`regression.md`

- 已确认产出目录与文件的命名规则：
  - **主题目录名**采用：`YYYY-MM-DD-<中文主题>/`
  - 其中日期表示**主题首次建立日期**，后续同主题继续维护原目录
  - **主文件名 / 阶段文件名**固定为：`current-truth.md`、`planning.md`、`regression.md`
  - 理由：目录名保留主题与时间上下文，文件名保持语义稳定，便于人工浏览和后续程序化处理

  - `问题`

  - 理由：既保持可读性，又避免 regression 文档退化成随意的问题记录或过重的 issue 管理模板
- 已将 `planning.md` 与 `regression.md` 的模板规则正式固化到 skill reference：
  - 新增 `references/stage-document-templates.md`
  - 在 `SKILL.md` 中补充了阶段文档模板的读取入口
  - 理由：把模板规则沉淀到 reference 层，避免继续压厚主 `SKILL.md`
- 新识别出的缺口：当前规则已经定义了“讨论文档如何存放”，但**还没有定义 discuss-and-distill 所管理的共享知识产物如何存放**。
  - 当前范围已经重新澄清：我们讨论的是 **`discuss-and-distill` 这个 skill 所管理的产出模型**，而不是整个 workspace 的通用知识库设计
  - 用户进一步说明：这里的“知识”可能包括**某个 API 的接口协议**、**某项技术的工作机制**等，这些内容会在后续讨论 current truth 时被直接引用
  - 因而当前尚未定义的是：在 `discuss-and-distill` 的产出体系内，这类**可跨主题复用的共享知识文档**应放在哪一层，以及它与主题目录、skill reference 的关系是什么
  - 理由：如果这层不定义，knowledge 类产物会在“主题 current truth”与“skill 自带 references”之间悬空；而它既不适合混入单个主题目录，也不完全等同于 skill 的内建规则说明
- 已确认一个新的边界条件：**共享知识层不需要预先规定具体文档格式**。
  - 理由：这类知识可能是 API 接口协议、技术工作机制、系统约束等，异质性很强；如果先强行统一模板，容易把 skill 产出模型设计成过重的知识管理系统
  - 当前更合适的做法是：先只定义**存放位置、产出角色、与主题 current truth 的关系**，把具体表达格式留给知识内容本身按需决定
- 已确认共享知识层**成立**。
  - 已确认需要同时支持两种知识作用域：
    - **主题内知识**：只在当前主题中使用或主要服务当前主题
    - **所有主题共享知识**：可被多个主题直接引用的知识
  - 这意味着 discuss-and-distill 的知识产物不能只有一层总 knowledge，还需要明确区分“主题内”和“跨主题共享”两个层级
- 已确认知识层与 current truth 的职责边界：
  - **`current-truth.md`**：承接当前主题下已经确认的结论、边界、判断与待决状态
  - **主题内知识 / 共享知识**：承接支撑这些判断的资料、协议说明、机制说明、上下文材料等可被引用的知识内容
  - 理由：避免把结论与依据混写，导致 `current-truth.md` 过厚、主题知识无处安放
- 已确认主题内知识与共享知识之间的流转关系：
  - 当主题内知识被确认可跨主题复用时，**允许晋升 / 复制**到 `docs-discuss-and-distill/knowledge/`
  - 当前只确认“允许流转”，**不强制规定固定流程**
  - 当前也不强制要求在晋升后删除原主题目录中的知识副本
- 已出现一轮新的结构改造提案，且它**会实质性改写之前的“主题 + planning/regression”模型**：
  - 当前倾向把“主题”改造为“**模块**”（名称仍可继续讨论，例如模块 / 组件）
  - 模块不是一次性讨论对象，而是更稳定的结构单元，并且**允许存在子模块**
  - 已确认：**submodule 不需要单独目录**，只需要作为模块目录内的独立子模块 truth 文件存在
  - 已确认：子模块 truth 采用**方案 B**，即独立轻文件，并统一使用 `submodule-` 前缀命名
  - 这意味着“子模块”是模块目录内的文件级单元，而不是目录级递归单元
  - 当前倾向把 `planning` 与 `regression` 从核心产物结构中移除，不再作为模块目录下的默认阶段文件
  - 当前倾向新增 `task` 概念：当基于某个模块发起一次代码落地时，抽离出一个独立 task，在 `docs-discuss-and-distill/` 下用 task 目录组织该次代码落地需要的流程与相关文件
  - 已限定当前第一类 task：**代码落地 task**
- 已确认新的顶层骨架采用四类顶层产物：
  - **模块树**：`docs-discuss-and-distill/modules/`
  - **主题树**：`docs-discuss-and-distill/topics/`
  - **task 树**：`docs-discuss-and-distill/tasks/`
  - **共享知识层**：`docs-discuss-and-distill/knowledge/`
  - 理由：模块承接软件模块型稳定对象，主题承接非软件模块型稳定对象，task 承接围绕对象发起的一次执行活动，共享知识层承接跨对象复用知识
- 已确认主题目录的最小形态：
  - `current-truth.md`
  - 理由：主题当前只承接一个 truth 文件，不引入模块式的 `knowledge/` 与 `submodule-*.md` 结构
- 已确认模块目录的最小形态：
  - `current-truth.md`
  - `knowledge/`
  - `submodule-*.md`
  - 其中 `submodule-*.md` 是模块内部的子模块 truth 文件，不是目录对象
- 已确认 `planning.md` 与 `regression.md` **先彻底退役**。
  - 当前它们不再属于核心模块结构，也不作为 task 的默认或可选文件
  - 理由：代码落地流程已被抽离为 task，对应产物应在新的 task 模型下重新定义，而不是沿用旧阶段文件
- 已确认旧的“主题目录 + planning/regression”规则已被**新模型替代**。
  - 理由：新的核心单位已经切换为模块，代码落地流程也已抽离到 task 层
- 已出现新的 task 体系想法，说明 task 不应只被理解为“代码落地目录”，而应被理解为**可承接不同执行意图的任务对象**。
  - 新出现的任务类型输入包括：
    - **直接创建任务**：任务可能先独立存在，且只与某个模块弱关联或暂时无关联；任务完成后，再把产出沉淀进对应模块与知识层
    - **基于某个子模块的代码落地任务**：任务可以直接围绕模块内部某个 `submodule-*.md` 展开
    - **面向某个模块的回归测试任务**：任务用于检查代码当前状态是否与 truth 相符合
  - 这意味着 task 不只是“执行代码落地”的容器，也可能是“先做任务，后沉淀模块 / 知识”的前置动作
- 已出现一个新的关键设计方向：**漂移处理（drift handling）**。
  - 用户希望 skill 具有鲁棒性，因此当回归测试发现“代码与 truth 不一致”时，需要有一套能承接漂移的处理逻辑
  - 当前尚未定义该逻辑，但它已经成为 task 体系设计中的关键输入
- 已出现一个新的稳定对象类别：**主题**。
  - 当前澄清：`modules/` 更偏向**软件模块**，用于承接明确的软件结构对象
  - 但仍存在一些讨论对象并不明确属于某个软件模块，因此需要一个独立的“主题”目录来承接这类讨论
  - 主题目录当前倾向只保留一个 truth 文件，不引入模块式的 `knowledge/` 与 `submodule-*.md` 结构
- 已确认主题与 task 可以建立关联关系：
  - 可以**从某个主题派生一个 task**
  - 也可以把**某个已完成 task 的结果沉淀回某个主题**
  - 这意味着 task 的挂接对象不应只局限于模块，也可能挂接到主题
- 这轮新输入的关键影响是：
  - “主题”不再是旧模型中的核心总单位
  - 但它仍然是新模型中**与模块并列存在的另一类稳定对象**，用于承接非软件模块型讨论
- 已确认 task 的核心模型：
  - **task 可以先独立创建**，不要求创建时就绑定某个模块或主题
  - task 后续可以**挂接到模块或主题**
  - task 完成后可以把结果**沉淀到模块 / 主题 / 知识层**
  - 这意味着 task 是一个相对独立的执行对象，而不是模块或主题的从属附属物
- 已确认第一版 task 类型集合至少包含三类：
  - **直接创建任务**
  - **基于子模块的代码落地任务**
  - **面向模块的回归测试任务**
  - 理由：这三类已经覆盖“先任务、后沉淀”“围绕子模块落地”“围绕 truth 做回归检查”三种核心执行意图
- 新识别到一个 task 建模上的混淆点：`直接创建任务` 与其余两类并不完全属于同一维度。
  - `直接创建任务` 更像**创建/挂接方式**：表示 task 可以先独立创建、后续再挂对象
  - `基于子模块的代码落地任务` 与 `面向模块的回归测试任务` 更像**任务意图/任务类型**
  - 因此在制定 task 命名规则前，可能需要先把“任务类型”与“任务挂接方式”拆成两个维度
- 已确认 drift handling 采用轻量三分法：
  - **truth 过时**：代码合理，但 truth 没有同步更新
  - **code 偏离**：truth 仍然成立，但代码跑偏了
  - **暂时无法判断**：当前证据不足，不能立即判断该改 truth 还是改 code
- 已确认三类 drift 的默认动作：
  - `truth 过时` → **更新 truth**，不直接改代码
  - `code 偏离` → **修正代码**，不直接改 truth
  - `暂时无法判断` → **挂起为待确认冲突**，不自动改 truth，也不自动宣判代码错误
- 已确认一条鲁棒性原则：
  - **先分类，再动作**
  - 在证据不足时，不自动回写 truth，也不自动修正代码

## 待确认项
- 当前工作术语采用：**模块**。
  - 当前还未最终拍板：是否保留“组件”等备选名字，还是正式固定为“模块”。
- 当前倾向把 task 模型拆成两个维度：**任务类型** 与 **任务挂接方式**。
  - 当前倾向：`直接创建任务` 更适合作为“挂接方式”，而不是和 `代码落地` / `回归测试` 并列的任务类型。

## 未决问题
- 当前最关键：task 是否应显式拆成“任务类型”与“任务挂接方式”两个维度。
- 其他：
  - 如果拆维度，第一版任务类型是否先收为 `代码落地` / `回归测试` 两类。
  - 如果拆维度，挂接方式是否至少区分：独立创建、挂模块、挂主题、挂子模块。
  - 所有主题共享知识层是否需要任何最小元信息（例如来源 / 更新时间），还是继续保持完全开放。
  - “模块”是否就是最终术语，还是后续还要在“模块 / 组件”之间再收一次口。

## 冲突与影响检查
- 原有“主题 + planning/regression”模型已被新的“模块 + task”模型替代，当前不再作为默认结构继续生效。
- 新出现一个需要重新校正的点：
  - 之前一度把“模块”理解为新的唯一稳定对象。
  - 现在用户补充：仍有不明确属于软件模块的讨论对象，因此需要把“主题”作为与模块并列的稳定对象重新引入。

- 新出现一个需要重新校正的点：
  - 之前曾把“当前第一类 task”收敛为代码落地 task。
  - 现在用户补充了更多 task 想法，说明这个判断应降级为**过渡性理解**，而不是最终 task 类型边界。
- 已识别到一个容易混淆的点：
  - “轻重分流成立” 与 “skill 要不要负责分流” 是两回事。
  - 当前已明确：前者成立，后者不成立。
- 已识别到一个关键设计点并已收敛：
  - “一致性复盘”不能缺席。
  - 在交互上默认显式保留，但允许按场景轻量化，以兼顾稳定性与自然度。
